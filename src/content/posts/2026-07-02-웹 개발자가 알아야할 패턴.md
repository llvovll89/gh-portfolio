웹 개발을 하다 보면 비슷한 문제를 반복해서 마주치게 된다. "이 상태를 어디에 둬야 하지?", "이 컴포넌트가 너무 많은 역할을 하고 있는 것 같은데?", "API 호출 실패를 어떻게 우아하게 처리하지?" 같은 고민들이다. 디자인 패턴은 이런 반복되는 문제에 대한 검증된 해결책이다. 패턴 자체를 외우는 것이 목적이 아니라, 패턴이 해결하려는 "문제 상황"을 이해하고 언제 적용해야 하는지 판단하는 능력이 중요하다.

> 목표: 웹 개발에서 자주 마주치는 상황별로 어떤 패턴을 적용할 수 있는지 파악하고, 실제 코드에 적용할 수 있는 수준까지 이해한다.

## 1. 왜 패턴을 알아야 하는가

디자인 패턴을 안다는 것은 단순히 코드를 예쁘게 짜는 기술이 아니다. 패턴은 여러 개발자가 오랜 시간에 걸쳐 검증한 "문제-해결 쌍"이기 때문에, 패턴을 알면 다음과 같은 이점이 있다.

- **커뮤니케이션 비용 감소**: "여기 Provider 패턴 써서 의존성 주입하자"라는 말 한마디로 설계 의도가 전달된다.
- **재발명 방지**: 이미 검증된 해결책이 있는데 처음부터 다시 고민할 필요가 없다.
- **유지보수성 향상**: 패턴을 따른 코드는 일정한 구조를 가지므로 다른 개발자가 이해하기 쉽다.

다만 패턴을 위한 패턴은 경계해야 한다. 문제가 없는데 패턴부터 적용하면 오히려 복잡도만 늘어난다. **"이 코드가 실제로 어떤 문제를 겪고 있는가"**를 먼저 파악하고 그에 맞는 패턴을 고르는 것이 순서다.

## 2. 컴포넌트 설계 패턴

### 2.1 Container / Presentational 패턴

로직과 뷰를 분리하는 가장 기본적인 패턴이다. Container는 데이터와 상태를 다루고, Presentational은 오직 UI 렌더링만 담당한다.

```tsx
// Container: 로직 담당
function UserListContainer() {
  const { users, isLoading } = useUsers();
  return <UserList users={users} isLoading={isLoading} />;
}

// Presentational: UI만 담당
function UserList({ users, isLoading }: UserListProps) {
  if (isLoading) return <Spinner />;
  return (
    <ul>
      {users.map((u) => (
        <li key={u.id}>{u.name}</li>
      ))}
    </ul>
  );
}
```

훅(Hook) 기반 개발이 자리잡으면서 이 패턴은 "커스텀 훅 + 순수 UI 컴포넌트" 형태로 진화했다. 로직은 `useUsers()` 같은 훅으로 빼고, 컴포넌트는 props만 받아서 그리는 방식이다.

### 2.2 Compound Component 패턴

여러 컴포넌트가 암묵적으로 상태를 공유하며 하나의 단위처럼 동작하는 패턴이다. `<select>`와 `<option>`의 관계를 떠올리면 이해가 쉽다.

```tsx
const TabsContext = createContext<TabsContextType | null>(null);

function Tabs({ children, defaultValue }: TabsProps) {
  const [active, setActive] = useState(defaultValue);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

function TabList({ children }: { children: ReactNode }) {
  return <div className="tab-list">{children}</div>;
}

function Tab({ value, children }: TabProps) {
  const { active, setActive } = useContext(TabsContext)!;
  return (
    <button
      className={active === value ? "active" : ""}
      onClick={() => setActive(value)}
    >
      {children}
    </button>
  );
}

// 사용
<Tabs defaultValue="profile">
  <TabList>
    <Tab value="profile">프로필</Tab>
    <Tab value="settings">설정</Tab>
  </TabList>
</Tabs>;
```

장점은 사용하는 쪽에서 마크업 순서와 구성을 자유롭게 조합할 수 있다는 점이다. `props drilling` 없이도 내부 컴포넌트끼리 상태를 공유한다.

### 2.3 Render Props / Custom Hook 패턴

로직을 재사용하는 두 가지 방식이다. Render Props는 함수를 자식으로 넘겨 렌더링을 위임하고, Custom Hook은 React Hooks가 등장한 이후 사실상 표준이 되었다.

```tsx
// Custom Hook 패턴 (현재 권장 방식)
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue((v) => !v), []);
  return [value, toggle] as const;
}

function Modal() {
  const [isOpen, toggle] = useToggle();
  return (
    <>
      <button onClick={toggle}>열기</button>
      {isOpen && <div className="modal">내용</div>}
    </>
  );
}
```

## 3. 상태 관리 패턴

| 패턴 | 적합한 상황 | 예시 도구 |
|------|------------|-----------|
| Local State | 컴포넌트 내부에서만 쓰이는 UI 상태 | `useState`, `useReducer` |
| Lifting State Up | 형제 컴포넌트 간 상태 공유 | React 기본 |
| Context + Reducer | 중간 규모의 전역 상태, prop drilling 회피 | `useContext` + `useReducer` |
| Store 기반 전역 상태 | 앱 전체에서 광범위하게 쓰이는 상태 | Zustand, Redux, Jotai |
| Server State | 서버에서 가져온 데이터의 캐싱/동기화 | React Query, SWR |

핵심은 **"모든 상태를 전역으로 두지 않는다"**는 원칙이다. 상태는 그것을 필요로 하는 컴포넌트에 최대한 가깝게 두고, 여러 곳에서 공유해야 할 때만 상위로 끌어올린다(Lifting State Up). 서버에서 가져온 데이터(Server State)와 클라이언트 UI 상태(Client State)를 구분해서 관리하는 것도 중요한 포인트다. 서버 상태는 React Query 같은 라이브러리로, UI 상태는 `useState`/Zustand 등으로 분리하면 코드가 훨씬 명확해진다.

```tsx
// Reducer 패턴: 복잡한 상태 전이 로직을 한 곳에 모은다
type Action =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "reset" };

function counterReducer(state: number, action: Action): number {
  switch (action.type) {
    case "increment":
      return state + 1;
    case "decrement":
      return state - 1;
    case "reset":
      return 0;
    default:
      return state;
  }
}
```

## 4. 데이터 페칭 패턴

### 4.1 Fetch-on-Render vs Fetch-then-Render

- **Fetch-on-Render**: 컴포넌트가 마운트된 후 `useEffect`로 데이터를 가져오는 방식. 구현은 쉽지만 워터폴(waterfall) 현상이 발생하기 쉽다.
- **Fetch-then-Render**: 라우트 진입 전에 필요한 데이터를 미리 가져오는 방식(React Router의 `loader`, Next.js의 서버 컴포넌트 등). 초기 렌더링 성능이 더 좋다.

### 4.2 캐싱 + 재검증(Stale-While-Revalidate) 패턴

React Query나 SWR이 채택한 패턴으로, 캐시된 데이터를 먼저 보여주고 백그라운드에서 최신 데이터를 다시 가져와 갱신한다.

```tsx
function usePosts() {
  return useQuery({
    queryKey: ["posts"],
    queryFn: fetchPosts,
    staleTime: 1000 * 60, // 1분간은 캐시를 신선하다고 간주
  });
}
```

이 패턴을 쓰면 사용자는 로딩 스피너를 자주 보지 않고, 데이터는 눈에 띄지 않게 최신 상태로 유지된다.

### 4.3 낙관적 업데이트(Optimistic Update)

서버 응답을 기다리지 않고 UI를 먼저 업데이트한 뒤, 실패하면 롤백하는 패턴이다. 좋아요 버튼, 토글 스위치처럼 즉각적인 피드백이 중요한 UI에 적합하다.

```tsx
function useLikePost() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: likePost,
    onMutate: async (postId) => {
      await queryClient.cancelQueries({ queryKey: ["post", postId] });
      const previous = queryClient.getQueryData(["post", postId]);
      queryClient.setQueryData(["post", postId], (old: Post) => ({
        ...old,
        liked: true,
      }));
      return { previous };
    },
    onError: (_err, postId, context) => {
      queryClient.setQueryData(["post", postId], context?.previous);
    },
  });
}
```

## 5. 비동기 처리 패턴

### 5.1 디바운스 / 쓰로틀

검색창 입력, 스크롤 이벤트처럼 짧은 시간에 여러 번 발생하는 이벤트를 제어하는 패턴이다.

```tsx
function useDebouncedValue<T>(value: T, delay = 300): T {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debounced;
}
```

- **디바운스**: 마지막 이벤트 이후 일정 시간이 지나야 실행 (검색어 자동완성)
- **쓰로틀**: 일정 주기마다 한 번만 실행 (스크롤 위치 감지)

### 5.2 큐(Queue) / 재시도(Retry) 패턴

네트워크가 불안정한 환경에서 요청을 순서대로 처리하거나 실패 시 자동으로 재시도하는 패턴이다.

```ts
async function fetchWithRetry(
  url: string,
  retries = 3,
  backoff = 500
): Promise<Response> {
  try {
    const res = await fetch(url);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return res;
  } catch (err) {
    if (retries === 0) throw err;
    await new Promise((r) => setTimeout(r, backoff));
    return fetchWithRetry(url, retries - 1, backoff * 2);
  }
}
```

## 6. 성능 최적화 패턴

- **메모이제이션(Memoization)**: `useMemo`, `useCallback`, `React.memo`로 불필요한 재계산/재렌더링 방지
- **코드 스플리팅(Code Splitting)**: `React.lazy` + `Suspense`로 초기 번들 크기 축소
- **가상화(Virtualization)**: 긴 리스트에서 화면에 보이는 항목만 렌더링 (`react-window`, `react-virtual`)
- **레이지 로딩(Lazy Loading)**: 이미지, 라우트 단위 지연 로딩

```tsx
const SettingsPage = lazy(() => import("./pages/SettingsPage"));

function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <SettingsPage />
    </Suspense>
  );
}
```

> 팁: 메모이제이션은 "측정 후 적용"이 원칙이다. React DevTools Profiler로 실제 병목을 확인하지 않고 무분별하게 `useMemo`를 남발하면, 오히려 메모리 오버헤드와 코드 복잡도만 늘어난다.

## 7. 백엔드 아키텍처 패턴

프론트엔드 개발자도 알아두면 협업에 도움이 되는 서버 사이드 패턴들이다.

| 패턴 | 설명 |
|------|------|
| Layered Architecture | Controller → Service → Repository로 계층을 나눠 책임을 분리 |
| Repository 패턴 | 데이터 접근 로직을 추상화해 DB 구현체를 쉽게 교체 |
| Middleware 패턴 | 요청/응답 파이프라인에 인증, 로깅 등 공통 로직을 끼워 넣음 |
| BFF (Backend for Frontend) | 프론트엔드 요구에 맞춰 API를 가공해주는 중간 계층 |
| Circuit Breaker | 장애가 발생한 외부 서비스 호출을 일정 시간 차단해 연쇄 장애 방지 |

```ts
// Express 미들웨어 패턴 예시
function authMiddleware(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization;
  if (!token) return res.status(401).json({ error: "Unauthorized" });
  next();
}

app.get("/api/profile", authMiddleware, getProfileHandler);
```

## 8. 에러 처리 패턴

### 8.1 Error Boundary

렌더링 중 발생한 예외를 특정 컴포넌트 트리 단위로 격리해서 앱 전체가 죽는 것을 방지한다.

```tsx
class ErrorBoundary extends Component<Props, State> {
  state: State = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error("에러 발생:", error, info);
  }

  render() {
    if (this.state.hasError) return <ErrorFallback />;
    return this.props.children;
  }
}
```

### 8.2 Result / Either 패턴

예외를 던지는 대신 성공/실패를 값으로 표현해서 타입 시스템이 에러 처리를 강제하도록 만드는 패턴이다.

```ts
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

async function fetchUser(id: string): Promise<Result<User>> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) return { ok: false, error: new Error("요청 실패") };
    return { ok: true, value: await res.json() };
  } catch (err) {
    return { ok: false, error: err as Error };
  }
}

// 사용하는 쪽에서 분기 처리를 빼먹기 어렵다
const result = await fetchUser("1");
if (result.ok) {
  console.log(result.value.name);
} else {
  console.error(result.error.message);
}
```

## 9. 정리 및 체크리스트

패턴을 도입하기 전에 스스로 던져볼 질문들이다.

- [ ] 지금 겪고 있는 문제가 실제로 이 패턴이 해결하는 문제인가?
- [ ] 이 패턴을 도입하면 팀원 모두가 이해할 수 있는가, 아니면 나만 아는 코드가 되는가?
- [ ] 패턴을 적용한 뒤 코드가 더 단순해졌는가, 아니면 레이어만 늘어났는가?
- [ ] 성능 최적화 패턴이라면, 실제로 병목을 측정했는가?

패턴은 도구일 뿐이다. 좋은 개발자는 패턴을 많이 아는 사람이 아니라, **문제 상황을 정확히 진단하고 그에 맞는 패턴을 적절한 시점에 적용하는 사람**이다. 이 글에서 다룬 패턴들을 실제 프로젝트에 하나씩 적용해보면서 "왜 이 패턴이 필요했는지"를 체감하는 것이 가장 좋은 학습 방법이다.
