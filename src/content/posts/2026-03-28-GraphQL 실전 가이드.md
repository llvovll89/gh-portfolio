GraphQL은 Facebook이 개발한 API 쿼리 언어로, 클라이언트가 **필요한 데이터만 정확히 요청**할 수 있습니다. REST의 오버페칭(불필요한 데이터 수신)과 언더페칭(여러 번 요청해야 하는 문제)을 해결하며, 단일 엔드포인트로 복잡한 데이터 관계를 효율적으로 다룹니다.

> 목표: GraphQL의 핵심 개념을 이해하고, 스키마 설계부터 서버 구현, React 클라이언트 연동까지 실무 수준의 GraphQL API를 직접 구축할 수 있다.

## 1. REST vs GraphQL — 언제 무엇을 쓸까

### 오버페칭 / 언더페칭 문제

```
# REST — 사용자 이름만 필요한데 전체 객체 수신 (오버페칭)
GET /api/users/1
Response: { id, name, email, phone, address, createdAt, updatedAt, ... }

# REST — 사용자 + 게시글 목록 = 2번 요청 (언더페칭)
GET /api/users/1
GET /api/users/1/posts

# GraphQL — 필요한 것만 한 번에
query {
  user(id: "1") {
    name
    posts {
      title
      createdAt
    }
  }
}
```

### 비교표

| 항목 | REST | GraphQL |
|------|------|---------|
| 엔드포인트 | 리소스별 다수 | 단일 (`/graphql`) |
| 데이터 형식 | 서버가 결정 | 클라이언트가 결정 |
| 버전 관리 | `/v1`, `/v2` 방식 | 스키마 진화 (deprecation) |
| 파일 업로드 | 자연스러움 | 추가 설정 필요 |
| 캐싱 | HTTP 캐시 기본 지원 | 별도 구현 필요 |
| 학습 곡선 | 낮음 | 중간 |

**GraphQL이 유리한 경우:**
- 클라이언트(앱, 웹)가 다양하고 필요한 데이터가 다를 때
- 복잡한 관계형 데이터 (소셜 그래프, 커머스)
- BFF(Backend for Frontend) 패턴

**REST가 유리한 경우:**
- 단순한 CRUD API
- 파일 업로드가 많을 때
- HTTP 캐싱이 중요할 때

## 2. GraphQL 핵심 개념

### 타입 시스템

GraphQL은 강타입 스키마를 기반으로 합니다.

```graphql
# 스칼라 타입: String, Int, Float, Boolean, ID
# 느낌표(!)는 Non-null (null 불허)

type User {
  id: ID!
  name: String!
  email: String!
  age: Int
  role: UserRole!
  posts: [Post!]!
  createdAt: String!
}

enum UserRole {
  ADMIN
  EDITOR
  VIEWER
}

type Post {
  id: ID!
  title: String!
  content: String!
  author: User!
  tags: [String!]!
  publishedAt: String
}
```

### 세 가지 작업

```graphql
# Query — 데이터 조회 (읽기)
query GetUser($id: ID!) {
  user(id: $id) {
    name
    email
  }
}

# Mutation — 데이터 변경 (쓰기)
mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
  }
}

# Subscription — 실시간 이벤트 (WebSocket)
subscription OnNewMessage($roomId: ID!) {
  newMessage(roomId: $roomId) {
    content
    sender { name }
  }
}
```

## 3. 스키마 설계 (Schema Definition Language)

### 입력 타입 (Input Types)

```graphql
# Mutation 인자는 Input 타입으로 묶음
input CreateUserInput {
  name: String!
  email: String!
  password: String!
  role: UserRole = VIEWER  # 기본값 설정
}

input UpdateUserInput {
  name: String
  email: String
  # id는 포함하지 않음 — 경로 파라미터로 받음
}

input PaginationInput {
  page: Int = 1
  limit: Int = 20
}
```

### 인터페이스와 유니온

```graphql
# 인터페이스 — 공통 필드 정의
interface Node {
  id: ID!
  createdAt: String!
}

type Article implements Node {
  id: ID!
  createdAt: String!
  title: String!
  body: String!
}

type Video implements Node {
  id: ID!
  createdAt: String!
  title: String!
  url: String!
  duration: Int!
}

# 유니온 — 여러 타입 중 하나
union SearchResult = Article | Video | User

type Query {
  search(term: String!): [SearchResult!]!
}
```

```graphql
# 클라이언트에서 유니온 타입 처리
query Search($term: String!) {
  search(term: $term) {
    ... on Article {
      title
      body
    }
    ... on Video {
      title
      url
      duration
    }
    ... on User {
      name
      email
    }
  }
}
```

### 페이지네이션 — Cursor-based (권장)

```graphql
type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}

type PostEdge {
  node: Post!
  cursor: String!
}

type PostConnection {
  edges: [PostEdge!]!
  pageInfo: PageInfo!
  totalCount: Int!
}

type Query {
  posts(
    first: Int
    after: String
    last: Int
    before: String
    filter: PostFilterInput
  ): PostConnection!
}
```

## 5. 리졸버 패턴과 컨텍스트

### 리졸버 구조

```ts
// src/resolvers.ts
import { Resolvers } from './generated/types'; // codegen 사용 시

export const resolvers: Resolvers = {
  Query: {
    users: async (_, __, { db }) => {
      return db.user.findMany();
    },

    user: async (_, { id }, { db }) => {
      return db.user.findUnique({ where: { id } });
    },

    posts: async (_, __, { db }) => {
      return db.post.findMany({ orderBy: { createdAt: 'desc' } });
    },
  },

  // 타입 리졸버 — 관계 필드 해결
  User: {
    posts: async (parent, _, { db }) => {
      // parent는 User 객체
      return db.post.findMany({
        where: { authorId: parent.id },
      });
    },
  },

  Post: {
    author: async (parent, _, { db }) => {
      return db.user.findUnique({ where: { id: parent.authorId } });
    },
  },

  Mutation: {
    createPost: async (_, { input }, { db, currentUser }) => {
      if (!currentUser) throw new GraphQLError('인증이 필요합니다.', {
        extensions: { code: 'UNAUTHENTICATED' },
      });

      return db.post.create({
        data: { ...input, authorId: currentUser.id },
      });
    },
  },
};
```

### 컨텍스트 — 인증 정보 공유

```ts
// src/context.ts
import { PrismaClient } from '@prisma/client';
import jwt from 'jsonwebtoken';

const db = new PrismaClient();

export async function createContext({ req }: { req: Request }) {
  const token = req.headers.get('authorization')?.replace('Bearer ', '');

  let currentUser = null;
  if (token) {
    try {
      const payload = jwt.verify(token, process.env.JWT_SECRET!) as { userId: string };
      currentUser = await db.user.findUnique({ where: { id: payload.userId } });
    } catch {
      // 유효하지 않은 토큰 — 비로그인 상태로 처리
    }
  }

  return { db, currentUser };
}

export type Context = Awaited<ReturnType<typeof createContext>>;
```

## 6. N+1 문제와 DataLoader

가장 흔한 GraphQL 성능 문제입니다. 100개의 Post를 조회하면 각 Post의 author를 위해 100번 추가 쿼리가 발생합니다.

```ts
// N+1 문제 발생 코드
Post: {
  author: async (parent, _, { db }) => {
    // Post가 100개면 → 100번 DB 쿼리!
    return db.user.findUnique({ where: { id: parent.authorId } });
  },
},
```

### DataLoader로 배치 처리

```bash
npm install dataloader
```

```ts
import DataLoader from 'dataloader';

// 여러 ID를 한 번에 로드하는 배치 함수
function createUserLoader(db: PrismaClient) {
  return new DataLoader<string, User>(async (userIds) => {
    const users = await db.user.findMany({
      where: { id: { in: [...userIds] } },
    });

    // DataLoader는 입력 순서대로 결과를 반환해야 함
    const userMap = new Map(users.map(u => [u.id, u]));
    return userIds.map(id => userMap.get(id) ?? new Error(`User ${id} not found`));
  });
}

// Context에 DataLoader 추가
export async function createContext({ req }) {
  return {
    db,
    currentUser,
    loaders: {
      user: createUserLoader(db), // 요청당 하나의 인스턴스
    },
  };
}

// 리졸버에서 사용
Post: {
  author: (parent, _, { loaders }) => {
    // 100개의 Post → 단 1번의 DB 쿼리로 배치 처리!
    return loaders.user.load(parent.authorId);
  },
},
```

## 7. Mutation — 데이터 변경

### 에러 처리 패턴

```graphql
# Union 기반 에러 처리 (권장)
type CreatePostSuccess {
  post: Post!
}

type ValidationError {
  field: String!
  message: String!
}

type AuthError {
  message: String!
}

union CreatePostResult = CreatePostSuccess | ValidationError | AuthError

type Mutation {
  createPost(input: CreatePostInput!): CreatePostResult!
}
```

```ts
// 리졸버
Mutation: {
  createPost: async (_, { input }, { db, currentUser }) => {
    if (!currentUser) {
      return { __typename: 'AuthError', message: '로그인이 필요합니다.' };
    }

    if (!input.title.trim()) {
      return {
        __typename: 'ValidationError',
        field: 'title',
        message: '제목을 입력해주세요.',
      };
    }

    const post = await db.post.create({
      data: { ...input, authorId: currentUser.id },
    });

    return { __typename: 'CreatePostSuccess', post };
  },
},
```

```graphql
# 클라이언트
mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    ... on CreatePostSuccess {
      post { id title }
    }
    ... on ValidationError {
      field
      message
    }
    ... on AuthError {
      message
    }
  }
}
```


```ts
// 스키마
const typeDefs = gql`
  type Subscription {
    messageAdded(roomId: ID!): Message!
    postLiked(postId: ID!): Post!
  }
`;

// 리졸버 — PubSub으로 이벤트 발행/구독
import { PubSub } from 'graphql-subscriptions';
const pubSub = new PubSub();

const resolvers = {
  Mutation: {
    sendMessage: async (_, { roomId, content }, { db, currentUser }) => {
      const message = await db.message.create({
        data: { roomId, content, senderId: currentUser.id },
        include: { sender: true },
      });

      // 구독자에게 이벤트 발행
      pubSub.publish(`MESSAGE_ADDED_${roomId}`, { messageAdded: message });
      return message;
    },
  },

  Subscription: {
    messageAdded: {
      subscribe: (_, { roomId }) =>
        pubSub.asyncIterator(`MESSAGE_ADDED_${roomId}`),
    },
  },
};
```

## 9. React 클라이언트 — Apollo Client

```bash
npm install @apollo/client graphql
```

```tsx
// src/main.tsx
import { ApolloClient, InMemoryCache, ApolloProvider, createHttpLink } from '@apollo/client';
import { setContext } from '@apollo/client/link/context';

const httpLink = createHttpLink({ uri: 'http://localhost:4000/graphql' });

const authLink = setContext((_, { headers }) => ({
  headers: {
    ...headers,
    authorization: localStorage.getItem('token')
      ? `Bearer ${localStorage.getItem('token')}`
      : '',
  },
}));

const client = new ApolloClient({
  link: authLink.concat(httpLink),
  cache: new InMemoryCache({
    typePolicies: {
      // ID 기반 캐시 정규화
      User: { keyFields: ['id'] },
      Post: { keyFields: ['id'] },
    },
  }),
});

function App() {
  return (
    <ApolloProvider client={client}>
      <Router />
    </ApolloProvider>
  );
}
```

### useQuery와 useMutation

```tsx
import { gql, useQuery, useMutation } from '@apollo/client';

const GET_POSTS = gql`
  query GetPosts {
    posts {
      id
      title
      author { name }
      createdAt
    }
  }
`;

const CREATE_POST = gql`
  mutation CreatePost($input: CreatePostInput!) {
    createPost(input: $input) {
      ... on CreatePostSuccess {
        post { id title author { name } createdAt }
      }
      ... on ValidationError {
        field
        message
      }
    }
  }
`;

function PostList() {
  const { data, loading, error } = useQuery(GET_POSTS, {
    fetchPolicy: 'cache-and-network', // 캐시 + 네트워크 병행
  });

  const [createPost, { loading: creating }] = useMutation(CREATE_POST, {
    // 새 포스트를 캐시에 직접 추가 (refetch 없이)
    update(cache, { data }) {
      const newPost = data?.createPost?.post;
      if (!newPost) return;

      cache.modify({
        fields: {
          posts(existingPosts = []) {
            const newPostRef = cache.writeFragment({
              data: newPost,
              fragment: gql`
                fragment NewPost on Post {
                  id title author { name } createdAt
                }
              `,
            });
            return [newPostRef, ...existingPosts];
          },
        },
      });
    },
  });

  if (loading) return <Spinner />;
  if (error) return <div>오류: {error.message}</div>;

  return (
    <div>
      <button
        onClick={() => createPost({ variables: { input: { title: '새 글', content: '내용' } } })}
        disabled={creating}
      >
        포스트 작성
      </button>
      <ul>
        {data?.posts.map(post => (
          <li key={post.id}>
            <strong>{post.title}</strong> — {post.author.name}
          </li>
        ))}
      </ul>
    </div>
  );
}
```

## 10. 성능 최적화와 보안

### 쿼리 깊이 제한 (Depth Limiting)

```bash
npm install graphql-depth-limit
```

```ts
import depthLimit from 'graphql-depth-limit';

const server = new ApolloServer({
  typeDefs,
  resolvers,
  validationRules: [
    depthLimit(7), // 최대 7 depth
  ],
});
```

### 쿼리 복잡도 제한 (Complexity Limiting)

```bash
npm install graphql-query-complexity
```

```ts
import { createComplexityLimitRule } from 'graphql-query-complexity';

const server = new ApolloServer({
  typeDefs,
  resolvers,
  validationRules: [
    createComplexityLimitRule(1000, {
      // 타입별 복잡도 점수
      scalarCost: 1,
      objectCost: 2,
      listFactor: 10,
    }),
  ],
});
```

### Persisted Queries — 프로덕션 보안

```ts
// 클라이언트는 쿼리 해시만 전송 → 임의 쿼리 실행 불가
import { createPersistedQueryLink } from '@apollo/client/link/persisted-queries';
import { sha256 } from 'crypto-hash';

const persistedQueriesLink = createPersistedQueryLink({ sha256 });

const client = new ApolloClient({
  link: persistedQueriesLink.concat(httpLink),
  cache: new InMemoryCache(),
});
```
