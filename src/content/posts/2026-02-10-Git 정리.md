# Git 정리 (현업에서 쓰는 Git 가이드)

**Git**은 분산형 버전 관리 시스템으로, 코드의 변경 이력을 추적하고 협업을 가능하게 하는 필수 도구입니다.

## 1) Git이란? (개념과 구조)

### 1.1 Git의 핵심 개념

**Git의 3가지 영역**:
```
작업 디렉토리 (Working Directory)
    ↓ git add
스테이징 영역 (Staging Area)
    ↓ git commit
로컬 저장소 (Local Repository)
    ↓ git push
원격 저장소 (Remote Repository)
```

- **Working Directory**: 실제 파일을 수정하는 작업 공간
- **Staging Area**: 커밋할 변경사항을 준비하는 임시 공간
- **Repository**: 변경 이력이 저장되는 공간 (로컬/원격)

### 1.2 왜 Git을 사용하나?

**장점**:
- 버전 관리: 언제든 이전 상태로 되돌릴 수 있음
- 협업: 여러 사람이 동시에 작업 가능
- 브랜치: 기능별로 독립적인 작업 환경 제공
- 백업: 원격 저장소에 코드 보관

## 2) 기본 설정 (최초 1회)

### 2.1 Git 설치 확인

```bash
git --version
# git version 2.40.0
```

### 2.2 사용자 정보 설정

```bash
# 전역 설정 (모든 프로젝트에 적용)
git config --global user.name "홍길동"
git config --global user.email "hong@example.com"

# 현재 프로젝트에만 적용
git config user.name "회사계정"
git config user.email "work@company.com"

# 설정 확인
git config --list
git config user.name
```

### 2.3 기본 에디터 설정 (선택사항)

```bash
# VS Code
git config --global core.editor "code --wait"

# Vim (기본값)
git config --global core.editor "vim"
```

### 2.4 줄바꿈 설정 (Windows/Mac 차이 해결)

```bash
# Windows
git config --global core.autocrlf true

# Mac/Linux
git config --global core.autocrlf input
```

### 3.2 현재 상태 확인

```bash
git status              # 가장 자주 쓰는 명령어!
```

## 4) 기본 워크플로우 (add → commit → push)

### 4.1 변경사항 추가 (Staging)

```bash
# 특정 파일만 스테이징
git add index.html
git add src/main.js

# 현재 폴더 모든 변경사항 스테이징
git add .

# 특정 확장자만
git add *.js

# 대화형으로 선택
git add -p              # 변경사항을 일부만 스테이징 가능
```

### 4.2 커밋 (변경사항 저장)

```bash
# 기본 커밋
git commit -m "feat: 로그인 기능 추가"

# 에디터로 긴 메시지 작성
git commit

# 스테이징 + 커밋 한번에 (추적 중인 파일만)
git commit -am "fix: 버그 수정"
```

**좋은 커밋 메시지 작성법**:
```bash
# Conventional Commits 규칙
feat: 새 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 포맷팅 (동작 변경 없음)
refactor: 코드 리팩토링
test: 테스트 추가
chore: 빌드/설정 변경

# 예시
git commit -m "feat: 사용자 프로필 페이지 추가"
git commit -m "fix: 로그인 시 null 에러 해결"
git commit -m "docs: README에 설치 가이드 추가"
```

### 4.3 원격 저장소에 업로드

```bash
# 최초 1회: 원격 저장소 연결
git remote add origin https://github.com/username/repo.git

# 푸시 (업로드)
git push origin main

# 기본 브랜치 설정 후 간단하게
git push -u origin main
git push                # 이후로는 이것만
```

### 4.4 원격 저장소에서 가져오기

```bash
# fetch + merge (안전)
git fetch origin        # 원격 변경사항 확인만
git merge origin/main   # 병합

# pull (fetch + merge 한번에)
git pull origin main
git pull                # 기본 브랜치 설정 후
```

## 5) 브랜치 관리

### 5.1 브랜치란?

브랜치는 **독립적인 작업 공간**입니다. `main`/`master`에 영향 없이 새 기능 개발 가능.

### 5.2 브랜치 기본 명령어

```bash
# 현재 브랜치 확인
git branch

# 새 브랜치 생성
git branch feature/login

# 브랜치 이동
git checkout feature/login

# 생성 + 이동 한번에 (자주 씀!)
git checkout -b feature/signup

# 최신 Git (2.23+)
git switch feature/login
git switch -c feature/new-feature

# 브랜치 삭제
git branch -d feature/login      # 병합된 브랜치만
git branch -D feature/login      # 강제 삭제
```

### 5.3 브랜치 병합 (Merge)

```bash
# 1) main으로 이동
git checkout main

# 2) 최신 상태로 업데이트
git pull

# 3) 브랜치 병합
git merge feature/login

# 4) 원격에 푸시
git push
```

### 5.4 충돌 해결

```bash
# 병합 시 충돌 발생
git merge feature/login
# CONFLICT (content): Merge conflict in index.html

# 1) 충돌 파일 확인
git status

# 2) 파일 열어서 수동 수정
# <<<<<<< HEAD
# 현재 브랜치 코드
# =======
# 병합하려는 브랜치 코드
# >>>>>>> feature/login

# 3) 수정 후 스테이징
git add index.html

# 4) 병합 완료
git commit              # 자동으로 merge 메시지 생성됨
```

### 5.5 Rebase (주의해서 사용)

```bash
# main의 최신 변경사항을 현재 브랜치에 적용
git checkout feature/login
git rebase main

# 충돌 시
git rebase --continue   # 수정 후 계속
git rebase --abort      # 취소
```

> **주의**: 이미 push한 커밋은 rebase 하지 마세요! (협업 시 문제 발생)

## 6) 원격 저장소 (GitHub/GitLab)

### 6.1 원격 저장소 관리

```bash
# 원격 저장소 확인
git remote -v

# 원격 저장소 추가
git remote add origin https://github.com/user/repo.git

# 원격 저장소 URL 변경
git remote set-url origin https://github.com/user/new-repo.git

# 원격 저장소 삭제
git remote remove origin
```

### 6.2 Pull Request (PR) 워크플로우

```bash
# 1) 이슈/티켓에서 브랜치 생성
git checkout -b feature/issue-123

# 2) 작업 후 커밋
git add .
git commit -m "feat: 이슈 #123 기능 구현"

# 3) 원격에 푸시
git push -u origin feature/issue-123

# 4) GitHub에서 PR 생성
# 5) 코드 리뷰 후 main에 병합
# 6) 로컬에서 main 업데이트
git checkout main
git pull
git branch -d feature/issue-123
```

## 7) 협업 워크플로우

### 7.1 Feature Branch 워크플로우

```bash
# 1) 항상 최신 main에서 시작
git checkout main
git pull

# 2) 기능 브랜치 생성
git checkout -b feature/user-profile

# 3) 작업 → 커밋
git add .
git commit -m "feat: 사용자 프로필 페이지 추가"

# 4) 원격에 푸시
git push -u origin feature/user-profile

# 5) Pull Request 생성 → 리뷰 → 병합

# 6) 병합 후 정리
git checkout main
git pull
git branch -d feature/user-profile
```

### 7.2 Git Flow (큰 프로젝트)

```
main (프로덕션)
  ↑
develop (개발)
  ↑
feature/* (기능 개발)
```

### 7.3 Fork & PR (오픈소스 기여)

```bash
# 1) GitHub에서 Fork 버튼 클릭

# 2) 내 저장소 클론
git clone https://github.com/my-username/repo.git
cd repo

# 3) 원본 저장소 연결
git remote add upstream https://github.com/original-owner/repo.git

# 4) 기능 브랜치 작성
git checkout -b fix/typo

# 5) 작업 후 내 저장소에 푸시
git push origin fix/typo

# 6) GitHub에서 원본 저장소로 PR 생성

# 7) 원본 저장소 최신 상태 동기화
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

## 8) 실수 복구 및 되돌리기

### 8.1 변경사항 취소

```bash
# Working Directory 변경사항 취소 (파일 복원)
git checkout -- index.html
git restore index.html          # 최신 Git

# Staging 취소 (add 취소)
git reset HEAD index.html
git restore --staged index.html # 최신 Git

# 모든 변경사항 취소 (주의!)
git reset --hard HEAD
```

### 8.2 커밋 수정

```bash
# 마지막 커밋 메시지만 수정
git commit --amend -m "fix: 올바른 커밋 메시지"

# 마지막 커밋에 파일 추가
git add forgot.txt
git commit --amend --no-edit    # 메시지 유지

# 주의: 이미 push한 커밋은 amend 하지 마세요!
```

### 8.3 커밋 되돌리기

```bash
# reset: 커밋 이력 삭제 (로컬 전용!)
git reset --soft HEAD~1         # 커밋만 취소, 파일 유지
git reset --mixed HEAD~1        # 커밋 + staging 취소, 파일 유지 (기본)
git reset --hard HEAD~1         # 커밋 + 파일 모두 삭제 (위험!)

# revert: 새 커밋으로 취소 (push 후에도 안전)
git revert HEAD                 # 마지막 커밋 취소
git revert a1b2c3d              # 특정 커밋 취소
```

### 8.4 실수로 삭제한 커밋 복구

```bash
# reflog 확인 (Git의 작업 이력)
git reflog

# 원하는 시점으로 복구
git reset --hard HEAD@{2}
```

### 8.5 브랜치 실수

```bash
# 잘못된 브랜치에서 작업했을 때
git stash                       # 변경사항 임시 저장
git checkout correct-branch     # 올바른 브랜치로 이동
git stash pop                   # 변경사항 복원

# 브랜치 이름 변경
git branch -m old-name new-name
```

## 9) .gitignore

### 9.1 .gitignore란?

Git이 **추적하지 않을 파일/폴더**를 지정하는 파일입니다.

### 9.2 자주 쓰는 패턴

```bash
# .gitignore 파일 예시

# 환경 변수
.env
.env.local
.env.production

# 의존성 (설치 파일)
node_modules/
vendor/
venv/
__pycache__/

# 빌드 결과물
dist/
build/
*.min.js
*.min.css

# IDE 설정
.vscode/
.idea/
*.swp
*.swo

# OS 생성 파일
.DS_Store
Thumbs.db
desktop.ini

# 로그 파일
*.log
logs/

# 임시 파일
*.tmp
*.bak
```

### 9.3 이미 추적 중인 파일 제거

```bash
# .gitignore에 추가했지만 이미 커밋된 파일
git rm --cached .env            # 파일은 유지, Git 추적만 중단
git rm -r --cached node_modules/

git commit -m "chore: .gitignore 적용"
git push
```

### 9.4 gitignore.io 활용

```bash
# 자동으로 .gitignore 생성
# https://www.toptal.com/developers/gitignore

# 예시: Node.js + React + VS Code
curl -L https://www.toptal.com/developers/gitignore/api/node,react,visualstudiocode > .gitignore
```

## 10) 유용한 명령어 치트시트

### 10.1 자주 쓰는 조합

| 목적 | 명령어 |
|---|---|
| 상태 확인 | `git status` |
| 변경사항 보기 | `git diff` |
| 커밋 이력 | `git log --oneline --graph` |
| 최신 업데이트 | `git pull` |
| 브랜치 생성 + 이동 | `git checkout -b feature/new` |
| 스테이징 + 커밋 | `git commit -am "메시지"` |
| 푸시 | `git push` |

### 10.2 커밋 이력 보기

```bash
# 기본 로그
git log

# 한 줄로 보기
git log --oneline

# 그래프로 보기
git log --oneline --graph --all

# 특정 파일 이력
git log -- index.html

# 특정 기간
git log --since="2 weeks ago"

# 특정 작성자
git log --author="홍길동"
```

### 10.3 변경사항 확인

```bash
# Working Directory vs Staging
git diff

# Staging vs 최신 커밋
git diff --staged
git diff --cached

# 브랜치 간 차이
git diff main..feature/login

# 특정 커밋 간 차이
git diff a1b2c3d..d4e5f6g
```

### 10.4 임시 저장 (Stash)

```bash
# 현재 작업 임시 저장
git stash
git stash save "작업 내용 메모"

# 저장 목록 확인
git stash list

# 복원
git stash pop               # 가장 최근 stash 적용 + 삭제
git stash apply             # 적용만 (삭제 안 함)
git stash apply stash@{1}   # 특정 stash 적용

# 삭제
git stash drop
git stash clear             # 전체 삭제
```

### 10.5 Alias (단축 명령어)

```bash
# 자주 쓰는 명령어 단축키 설정
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.cm commit
git config --global alias.lg "log --oneline --graph --all"

# 사용
git st                      # git status
git co main                 # git checkout main
git lg                      # 예쁜 로그
```

### 10.6 검색

```bash
# 커밋 메시지에서 검색
git log --grep="로그인"

# 코드에서 검색
git grep "TODO"

# 특정 문자열이 추가/삭제된 커밋 찾기
git log -S "function login"
```

## 마무리

Git은 **처음엔 어렵지만, 익숙해지면 필수 도구**입니다.

**핵심 요약**:
1. **기본 워크플로우**: `add → commit → push`
2. **브랜치**: 독립적인 작업 환경, `checkout -b`로 생성
3. **협업**: Pull Request 활용, main 항상 최신 유지
4. **실수 복구**: `reset` (로컬), `revert` (원격 push 후)
5. **.gitignore**: 민감 정보/빌드 결과물 제외

**실수 방지 팁**:
- 커밋 전 `git status`로 확인
- push 전 `git log`로 이력 확인
- main 브랜치에서 직접 작업 금지 (브랜치 사용!)
- 중요한 작업 전 `git stash` 또는 커밋으로 보존

**다음 단계**:
- GitHub Actions로 CI/CD 자동화
- Git Hooks로 커밋 전 자동 검증
- Cherry-pick, Bisect 등 고급 명령어 학습
- 팀의 Git 컨벤션 정립
