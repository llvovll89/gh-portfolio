드론을 활용한 순찰·안심동행 서비스처럼 **실시간성과 안전이 동시에 요구되는 시스템**을 개발할 때는, 일반 웹 서비스와는 다른 설계 고려 사항이 많습니다. 이 글은 실제 개발 회의에서 도출된 수정사항들을 토대로, 그 작업을 수행할 때 꼭 알아야 할 기술적 지식을 체계적으로 정리한 실전 가이드입니다.

> **목표:** 드론 기반 서비스(순찰, 안심동행 등)를 개발하거나 수정할 때 필요한 핵심 개념과 패턴을 익히고, 실수 없이 작업을 진행할 수 있는 기반 지식을 갖춘다.

## 1. 기능 비활성화 전략

기능을 "삭제"하는 것과 "진입 차단"하는 것은 전혀 다른 접근입니다. 회의에서 경찰 순찰 기능, 경로 모니터링, 길 안내 기능을 보류하기로 결정했지만 **코드를 삭제하지 않고 진입 차단 방식**을 선택한 이유가 있습니다.

### 왜 진입 차단을 선택하는가?

| 방식 | 장점 | 단점 |
|------|------|------|
| 코드 삭제 | 코드베이스 정리, 번들 크기 감소 | 복구 어려움, git 이력 의존도 높아짐 |
| 진입 차단 | 빠른 롤백, 다음 회의 후 결정 유연 | 데드 코드 증가 |

### 프론트엔드 라우트 비활성화 패턴

```tsx
// src/app/routes.tsx
const routes = [
  {
    path: '/patrol',
    element: <PatrolPage />,
  },
  // 경찰 기능 보류 — 진입 차단
  {
    path: '/police-patrol',
    element: <ComingSoon message="준비 중인 기능입니다." />,
    // element: <PolicePatrolPage />,  // 원본 유지
  },
];
```

### 사이드바 메뉴 숨김 처리

```tsx
const NAV_ITEMS = [
  { label: '수요 순찰', path: '/patrol', visible: true },
  { label: '안심동행', path: '/escort', visible: true },
  { label: '경찰 순찰', path: '/police-patrol', visible: false }, // 보류
];

// 렌더링 시
{NAV_ITEMS.filter(item => item.visible).map(item => (
  <NavItem key={item.path} {...item} />
))}
```

> **팁:** `visible` 플래그를 환경변수나 feature flag 시스템으로 관리하면, 배포 없이 기능 노출 여부를 조정할 수 있습니다.

## 2. DB 마이그레이션

운영 중인 서비스의 DB를 변경할 때는 항상 **하위 호환성과 롤백 전략**을 먼저 고려해야 합니다.

### 설정값 변경 (UPDATE)

운영시간처럼 단순 값 변경은 `UPDATE`로 처리하지만, 반드시 트랜잭션 안에서 실행하고 변경 전 값을 백업해두어야 합니다.

```sql
-- 변경 전 백업 (선택)
CREATE TABLE patrol_settings_backup AS SELECT * FROM patrol_settings;

-- 운영시간 변경
BEGIN;
UPDATE patrol_settings
SET operating_start = '22:00',
    operating_end   = '24:00';
COMMIT;
```

### 컬럼 추가 (ALTER TABLE)

새 필드 추가 시 `DEFAULT` 값을 명시하면 기존 데이터와의 호환이 유지됩니다.

```sql
-- 비상연락처 필드 추가
ALTER TABLE user_details
  ADD COLUMN emergency_contact_name  VARCHAR(100),
  ADD COLUMN emergency_contact_phone VARCHAR(20);

-- 알림 설정 필드 (기본값 ON)
ALTER TABLE user_details
  ADD COLUMN patrol_notification_enabled BOOLEAN NOT NULL DEFAULT TRUE;
```

### Spring Boot + Flyway/Liquibase 사용 시

```
resources/db/migration/
  V20260522__add_emergency_contact.sql
  V20260522__update_patrol_operating_hours.sql
```

파일명 규칙: `V{날짜}__{설명}.sql` — 버전 충돌 방지를 위해 날짜+시퀀스를 사용하는 팀도 많습니다.

> **주의:** 운영 DB에서 `ALTER TABLE`은 테이블 크기에 따라 락이 걸릴 수 있습니다. 대용량 테이블이라면 `pt-online-schema-change` 같은 도구를 검토하세요.

## 3. 배치 스케줄링과 소티 관리

드론 순찰에서 "소티(Sortie)"란 드론이 이륙해서 복귀하기까지 한 번의 비행 단위입니다. **1 소티 = 20분, 20가구 전체 순찰**이라는 제약 조건을 코드에 어떻게 반영하는지가 핵심입니다.

### 방문 시각 산출 공식

```java
// PatrolBatchService.java
private static final int SORTIE_DURATION_MINUTES = 20;
private static final int TOTAL_HOUSEHOLDS = 20;

// 가구당 평균 방문 시간: 20분 / 20가구 = 1분
private static final double MINUTES_PER_POINT =
    (double) SORTIE_DURATION_MINUTES / TOTAL_HOUSEHOLDS; // 1.0

public LocalDateTime estimateVisitTime(LocalDateTime sortieStart, int pointIndex) {
    long offsetSeconds = (long)(pointIndex * MINUTES_PER_POINT * 60);
    return sortieStart.plusSeconds(offsetSeconds);
}
```

### 운영 시간 기반 스케줄러

```java
// PatrolBatchScheduler.java
@Scheduled(cron = "0 0 22 * * *") // 매일 22:00
public void startPatrol() {
    LocalTime now = LocalTime.now();
    LocalTime end = LocalTime.of(0, 0); // 24:00 = 자정

    patrolBatchService.startSortie();
}

// application.yml
patrol:
  operating-start: "22:00"
  operating-end: "24:00"
  sortie-duration-minutes: 20
```

> **포인트:** 상수를 하드코딩하지 말고 `application.yml`로 외부화하면 재배포 없이 운영 파라미터를 조정할 수 있습니다.

## 4. 예약 시스템 설계

### 예약 마감 로직 (T-60)

```java
// PatrolReservationService.java
public ReservationResult reserve(ReservationRequest req, User user) {
    LocalDateTime slotStart = req.getSlotStart();

    // T-60: 운행 1시간 전까지만 신청 가능
    if (!LocalDateTime.now().isBefore(slotStart.minusMinutes(60))) {
        return saveRejected(req, user,
            "운행 1시간 전까지만 신청할 수 있습니다.", slot);
    }

    // 중복 시간대 신청 불가
    boolean alreadyBooked = reservationRepository
        .existsBySlotStartAndStatus(slotStart, ReservationStatus.CONFIRMED);
    if (alreadyBooked) {
        return saveRejected(req, user, "이미 예약된 시간대입니다.", slot);
    }

    return saveConfirmed(req, user, slot);
}
```

### 프론트엔드에서 예약 불가 시간대 표시

```tsx
interface TimeSlot {
  time: string;
  available: boolean;
}

function TimeSlotPicker({ slots }: { slots: TimeSlot[] }) {
  return (
    <div className="grid grid-cols-4 gap-2">
      {slots.map(slot => (
        <button
          key={slot.time}
          disabled={!slot.available}
          className={slot.available
            ? 'bg-blue-500 text-white'
            : 'bg-gray-200 text-gray-400 cursor-not-allowed line-through'
          }
        >
          {slot.time}

      ))}

```

> **UX 팁:** 비활성화된 슬롯에는 "예약됨" 툴팁을 추가하면 사용자가 이유를 알 수 있어 혼란을 줄입니다.

## 5. 긴급 상황 처리 패턴

긴급 순찰과 SOS처럼 **진행 중인 작업을 중단하고 우선 처리 후 재개**해야 하는 패턴은 상태 저장이 핵심입니다.

### 소티 중단 지점 저장

```java
// PatrolBatch.java (엔티티)
@Entity
public class PatrolBatch {
    @Id
    private Long id;

    private int lastCompletedPointIndex; // 마지막 완료 포인트 인덱스
    private PatrolBatchStatus status;    // RUNNING, INTERRUPTED, COMPLETED
    private LocalDateTime interruptedAt;
}
```

### 긴급 순찰 서비스 흐름

```java
// PatrolEmergencyService.java (신규)
@Service
@Transactional
public class PatrolEmergencyService {

    public void handleEmergency(Long batchId, EmergencyRequest req) {
        PatrolBatch batch = patrolBatchRepository.findById(batchId)
            .orElseThrow();

        // 1. 소티 중단 지점 저장
        batch.setStatus(PatrolBatchStatus.INTERRUPTED);
        batch.setInterruptedAt(LocalDateTime.now());
        patrolBatchRepository.save(batch);

        // 2. 드론 해당 가구 이동 + 호버링
        droneControlClient.moveToLocation(req.getHouseholdCoord());
        droneControlClient.startHovering();

        // 3. SMS 발송 (관계자 + 사용자 지정 연락처)
        smsService.sendEmergencyAlert(req.getUserId(),
            req.getStreamingLink(), req.getAddress());

        // 4. 관제 알림
        notificationService.notifyAdmins(EmergencyEvent.of(batchId, req));
    }

    public void resumeSortie(Long batchId) {
        PatrolBatch batch = patrolBatchRepository.findById(batchId)
            .orElseThrow();

        int resumeFrom = batch.getLastCompletedPointIndex() + 1;
        batch.setStatus(PatrolBatchStatus.RUNNING);
        patrolBatchService.continueSortie(batch, resumeFrom);
    }
}
```

### 상태 머신(State Machine) 관점으로 이해하기

```
PENDING → RUNNING → INTERRUPTED → RUNNING (재개)
                  ↘ COMPLETED
```

복잡한 드론 제어 로직일수록 상태를 명확히 정의하고, 각 전이(transition) 조건을 코드에 명시하는 것이 버그를 예방합니다.

## 6. 실시간 알림 구현

관리자 웹에서 긴급 순찰, SOS, 문자 발송 이력을 실시간으로 모니터링해야 할 때 선택지는 두 가지입니다.

### WebSocket vs SSE 비교

| 항목 | WebSocket | SSE (Server-Sent Events) |
|------|-----------|--------------------------|
| 통신 방향 | 양방향 | 서버 → 클라이언트 (단방향) |
| 복잡도 | 높음 | 낮음 |
| 자동 재연결 | 수동 구현 필요 | 브라우저 기본 지원 |
| 적합한 경우 | 채팅, 협업 | 대시보드 알림, 로그 스트리밍 |

**모니터링 대시보드처럼 서버→클라이언트 단방향 알림이면 SSE가 더 단순합니다.**

### Spring Boot SSE 구현

```java
// AdminNotificationController.java
@GetMapping(value = "/admin/events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter streamEvents() {
    SseEmitter emitter = new SseEmitter(Long.MAX_VALUE);
    sseEmitterRegistry.register(emitter);
    return emitter;
}

// NotificationService.java
public void notifyAdmins(EmergencyEvent event) {
    sseEmitterRegistry.getAll().forEach(emitter -> {
        try {
            emitter.send(SseEmitter.event()
                .name("emergency")
                .data(objectMapper.writeValueAsString(event)));
        } catch (IOException e) {
            emitter.complete();
        }
    });
}
```

### 프론트엔드 SSE 수신

```tsx
useEffect(() => {
  const es = new EventSource('/admin/events');

  es.addEventListener('emergency', (e) => {
    const event = JSON.parse(e.data);
    setEmergencyAlerts(prev => [event, ...prev]);
  });

  return () => es.close();
}, []);
```

## 7. SMS 발송 연동

긴급 순찰과 SOS 양쪽에서 동일한 SMS 인프라를 재사용하는 패턴입니다.

### 메시지 구성 원칙

```java
// NcpSmsService.java
public void sendEmergencyAlert(Long userId, String streamingLink, String address) {
    UserDetail user = userDetailRepository.findByUserId(userId);

    // 발송 대상: 시스템 관계자 + 사용자 지정 연락처
    List<String> recipients = new ArrayList<>(systemContacts);
    if (user.getEmergencyContactPhone() != null) {
        recipients.add(user.getEmergencyContactPhone());
    }

    String message = String.format(
        "[긴급알림] %s 주소에서 긴급 순찰이 요청되었습니다.%n" +
        "실시간 영상: %s",
        address, streamingLink
    );

    recipients.forEach(phone -> smsClient.send(phone, message));
}
```

### 발송 이력 저장

```java
@Entity
public class SmsLog {
    private Long id;
    private String recipient;
    private String messageType; // EMERGENCY_PATROL, SOS
    private String content;
    private SmsStatus status;   // SENT, FAILED
    private LocalDateTime sentAt;
}
```

> **중요:** SMS 발송은 실패할 수 있습니다. 발송 결과를 반드시 DB에 기록하고, 실패 시 재시도 로직(Retry)이나 알림을 갖추어야 합니다.

## 9. 용어 일괄 변경 작업

"긴급출동" → "SOS"처럼 **전체 코드베이스에서 용어를 바꾸는 작업**은 생각보다 범위가 넓습니다.

### 영향 범위 체크리스트

- [ ] UI 텍스트 (JSX, i18n 키)
- [ ] API 응답 필드명 (하위 호환 고려 필요)
- [ ] DB 컬럼명 / Enum 값
- [ ] 서비스/메서드명 (`handleEmergency` → `handleSOS`)
- [ ] API 엔드포인트 경로
- [ ] 로그 메시지
- [ ] 테스트 코드

### 안전한 방식: 별칭(Alias) 처리

API 필드명을 즉시 바꾸면 앱 클라이언트가 깨질 수 있습니다. 단계적으로 처리하세요.

```java
// 기존 필드 유지 + 새 필드 추가 → 이후 기존 필드 제거
public class EscortResponse {
    @Deprecated
    private String emergencyStatus; // 구버전 클라이언트 호환

    private String sosStatus;       // 신규 필드
}
```

### 프론트엔드 i18n 키 변경

```json
// ko.json
{
  "escort": {
    "sos": "SOS",
    "sosDescription": "SOS 요청 시 드론이 즉시 출동하여 기체 하강, 경보음, 서치라이트를 작동합니다."
  }
}
```

## 10. 작업 우선순위 결정 원칙

여러 수정사항이 동시에 쏟아질 때, 작업 순서를 잘못 잡으면 의존성 충돌이 발생합니다.

### 의존성 기반 순서 결정

```
[독립] 설정값 변경 (운영시간, 예약 마감)
  ↓
[독립] UI 비활성화 / 텍스트 제거
  ↓
[독립] DB 마이그레이션 (비상연락처, 알림 설정 필드 추가)
  ↓
[의존] 예약 충돌 방지 로직 (DB 준비 후 가능)
  ↓
[의존] SOS 기능 (용어 + 즉시 실행 = 함께 배포)
  ↓
[가장 복잡] 긴급 순찰 소티 중단/재개 (신규 엔티티 + 서비스 + API)
  ↓
[마지막] 관리자 모니터링 대시보드 (위 기능들 완료 후)
```

### 원칙 요약

1. **독립적인 작업 먼저** — 다른 작업에 영향을 주지 않는 것부터
2. **DB 마이그레이션은 코드 배포보다 먼저** — 새 컬럼이 없으면 서비스 오류 발생
3. **용어 변경 + 기능 변경은 함께** — SOS 용어만 바꾸고 기능이 없으면 사용자 혼란
4. **복잡한 신규 로직은 후순위** — 안정적인 기반이 먼저 있어야 테스트 가능
5. **미확정 사항은 작업하지 않는다** — 호버링 지속 시간 등은 회의 후 확정 전까지 보류

## 마무리

드론 서비스처럼 **물리 장치와 연동되는 시스템**은 일반 웹 서비스보다 훨씬 많은 상태 전이와 예외 상황을 다뤄야 합니다. 이 글에서 다룬 패턴들을 정리하면:

| 주제 | 핵심 포인트 |
|------|-----------|
| 기능 비활성화 | 삭제보다 진입 차단, 빠른 롤백 가능 |
| DB 마이그레이션 | DEFAULT 값 명시, 버전 관리 필수 |
| 배치 스케줄링 | 상수 외부화, 방문 시각 공식 명확히 |
| 예약 시스템 | 마감 시간 + 중복 체크 동시 적용 |
| 긴급 상황 | 중단 지점 저장 → 재개 가능 구조 |
| 실시간 알림 | 단방향이면 SSE가 단순하고 충분 |
| SMS 발송 | 이력 저장 + 실패 처리 필수 |
| 용어 변경 | 영향 범위 전체 체크 + 단계적 전환 |
| 작업 순서 | 의존성 분석 후 독립 작업부터 |

회의 한 번에 나온 수정사항도 이렇게 다양한 기술 영역에 걸쳐있습니다. 각 영역의 기본기를 탄탄히 다져두면 복잡해 보이는 작업도 하나씩 침착하게 풀어나갈 수 있습니다.
