> **목표:** 불필요한 리렌더링을 줄이고, 번들 크기를 최소화하며, 사용자에게 더 빠른 인터랙션을 제공한다.

## 1. 리렌더링 이해하기

React는 다음 세 가지 경우에 컴포넌트를 리렌더링한다.

- **state**가 변경될 때
- **props**가 변경될 때
- **부모 컴포넌트**가 리렌더링될 때

문제는 세 번째 경우다. 부모가 리렌더링되면 자식 컴포넌트는 props가 바뀌지 않아도 기본적으로 전부 다시 렌더링된다. 이게 쌓이면 심각한 성능 저하로 이어진다.

```tsx
function Parent() {
  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>+</button>
      {/* count와 무관하지만 부모가 리렌더링될 때마다 같이 렌더링된다 */}
      <HeavyChild />
    </>
  );
}
```

## 2. React.memo로 컴포넌트 메모이제이션

`React.memo`는 props가 변경되지 않으면 이전 렌더링 결과를 재사용한다. 얕은 비교(shallow comparison)를 사용하기 때문에 원시값과 안정적인 참조에 효과적이다.

```tsx
const HeavyChild = React.memo(function HeavyChild({ title }: { title: string }) {
  console.log('HeavyChild 렌더링');
  return <div>{title}</div>;
});
```

### 커스텀 비교 함수

객체나 배열 props를 비교할 때는 두 번째 인자로 비교 함수를 전달한다.

```tsx
const Chart = React.memo(
  function Chart({ data }: { data: number[] }) {
    return <canvas />;
  },
  (prev, next) => prev.data.length === next.data.length
);
```

> **주의:** `React.memo`는 남발하면 오히려 독이다. 비교 비용이 렌더링 비용보다 클 수 있다. props가 자주 바뀌는 컴포넌트엔 적용해도 효과가 없다.

## 3. useMemo — 값 메모이제이션

계산 비용이 높은 값을 의존성이 변경될 때만 재계산한다.

```tsx
function ProductList({ products, filterText }: Props) {
  // filterText나 products가 바뀔 때만 필터링 재실행
  const filtered = useMemo(
    () => products.filter(p => p.name.includes(filterText)),
    [products, filterText]
  );

  return <ul>{filtered.map(p => <li key={p.id}>{p.name}</li>)}</ul>;
}
```

### 언제 쓸까?

| 상황 | useMemo 필요 여부 |
|------|-----------------|
| 단순 문자열 가공 | ❌ 불필요 |
| 대용량 배열 필터링/정렬 | ✅ 권장 |
| 자식에게 넘기는 객체/배열 참조 유지 | ✅ 권장 |
| 렌더링 중 실행되는 무거운 연산 | ✅ 권장 |

## 4. useCallback — 함수 메모이제이션

함수를 의존성이 바뀔 때만 새로 생성한다. `React.memo`로 감싼 자식에 콜백을 넘길 때 특히 중요하다. 매 렌더링마다 새 함수 참조가 생기면 memo가 의미 없어지기 때문이다.

```tsx
function Parent() {
  const [items, setItems] = useState<string[]>([]);

  // useCallback 없이 쓰면 렌더링마다 새 함수 → Child 리렌더링
  const handleAdd = useCallback((item: string) => {
    setItems(prev => [...prev, item]);
  }, []); // 의존성 없음 — 최초 1회만 생성

  return <Child onAdd={handleAdd} />;
}

const Child = React.memo(function Child({ onAdd }: { onAdd: (item: string) => void }) {
  return <button onClick={() => onAdd('새 항목')}>추가</button>;
});
```

## 5. 상태 구조 최적화

### 상태 끌어내리기 (State Colocation)

상태를 실제로 필요한 컴포넌트 가까이 두면 불필요한 상위 리렌더링을 막을 수 있다.

```tsx
// ❌ 나쁜 예 — count 변경이 HeavySection까지 리렌더링
function App() {
  const [count, setCount] = useState(0);
  return (
    <>
      <Counter count={count} setCount={setCount} />
      <HeavySection />
    </>
  );
}

// ✅ 좋은 예 — Counter 자체가 상태를 소유
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

function App() {
  return (
    <>
      <Counter />
      <HeavySection />
    </>
  );
}
```

### children prop으로 슬롯 분리

```tsx
function Layout({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(false);
  // open이 바뀌어도 children은 리렌더링되지 않는다
  return (
    <div>
      <button onClick={() => setOpen(o => !o)}>토글</button>
      {open && <Sidebar />}
      {children}
    </div>
  );
}
```

## 6. 코드 스플리팅과 Lazy Loading

초기 번들에 모든 페이지를 포함하면 첫 로드가 느려진다. `React.lazy`와 `Suspense`로 필요할 때만 불러온다.

```tsx
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

const Home = lazy(() => import('./pages/Home'));
const Blog = lazy(() => import('./pages/Blog'));
const Projects = lazy(() => import('./pages/Projects'));

function App() {
  return (
    <Suspense fallback={<div className="loading-spinner" />}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/blog" element={<Blog />} />
        <Route path="/projects" element={<Projects />} />
      </Routes>
    </Suspense>
  );
}
```

### 컴포넌트 단위 Lazy Loading

무거운 라이브러리를 쓰는 컴포넌트도 분리할 수 있다.

```tsx
// 차트 라이브러리는 해당 페이지에 접근할 때만 로드
const HeavyChart = lazy(() => import('./components/HeavyChart'));
```

## 7. 가상화(Virtualization)로 긴 리스트 처리

수천 개의 DOM 노드를 한 번에 렌더링하면 브라우저가 버틸 수 없다. 가상화는 화면에 보이는 항목만 렌더링한다.

```bash
npm install @tanstack/react-virtual
```

```tsx
import { useVirtualizer } from '@tanstack/react-virtual';

function VirtualList({ items }: { items: string[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const rowVirtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 48,
  });

  return (
    <div ref={parentRef} style={{ height: '500px', overflow: 'auto' }}>
      <div style={{ height: rowVirtualizer.getTotalSize() + 'px', position: 'relative' }}>
        {rowVirtualizer.getVirtualItems().map(virtualRow => (
          <div
            key={virtualRow.index}
            style={{
              position: 'absolute',
              top: virtualRow.start + 'px',
              height: virtualRow.size + 'px',
              width: '100%',
            }}
          >
            {items[virtualRow.index]}
          </div>
        ))}
      </div>
    </div>
  );
}
```

10,000개 항목도 60fps를 유지할 수 있다.

## 8. useTransition과 useDeferredValue

React 18에서 추가된 동시성(Concurrency) API로, 급하지 않은 업데이트를 뒤로 미룰 수 있다.

### useTransition

```tsx
function SearchPage() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState<string[]>([]);
  const [isPending, startTransition] = useTransition();

  function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
    setQuery(e.target.value);

    // 검색 결과 업데이트는 낮은 우선순위로 처리
    startTransition(() => {
      setResults(heavySearch(e.target.value));
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending ? <Spinner /> : <ResultList results={results} />}
    </>
  );
}
```

### useDeferredValue

외부에서 내려오는 값을 디퍼링할 때 사용한다.

```tsx
function App() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {/* deferredQuery는 query보다 늦게 반영 — 입력이 막히지 않는다 */}
      <HeavyList query={deferredQuery} />
    </>
  );
}
```

## 9. 번들 사이즈 최적화

### Tree Shaking 활용

named import를 사용하면 번들러가 필요한 것만 포함한다.

```tsx
// ❌ 전체 lodash를 번들에 포함
import _ from 'lodash';

// ✅ debounce만 포함
import debounce from 'lodash/debounce';
// 또는
import { debounce } from 'lodash-es';
```

### Dynamic Import로 무거운 의존성 분리

```tsx
async function exportToPDF() {
  // 버튼 클릭 시에만 jsPDF 로드
  const { jsPDF } = await import('jspdf');
  const doc = new jsPDF();
  doc.save('export.pdf');
}
```

## 마무리

최적화는 **측정 → 발견 → 수정** 순서로 진행해야 한다. 프로파일링도 하기 전에 `useMemo`와 `useCallback`을 코드 전체에 뿌리는 건 오히려 코드 복잡도만 높인다.

| 기법 | 효과 | 적용 난이도 |
|------|------|------------|
| React.memo | 불필요한 리렌더링 차단 | ★★☆ |
| useMemo / useCallback | 계산·함수 재사용 | ★★☆ |
| 상태 끌어내리기 | 리렌더링 범위 축소 | ★☆☆ |
| Lazy Loading | 초기 번들 분리 | ★☆☆ |
| 가상화 | 대용량 리스트 처리 | ★★★ |
| useTransition | UI 응답성 유지 | ★★☆ |
| Tree Shaking | 번들 사이즈 감소 | ★☆☆ |

실무에서 가장 먼저 체감 효과가 큰 건 **코드 스플리팅**과 **상태 끌어내리기**다. 이 두 가지만 제대로 적용해도 앱이 눈에 띄게 가벼워진다.
