ChatGPT에게 함수 하나 짜달라고 했다. 코드가 나왔다. 그럴듯해 보인다. 그냥 복붙해도 될까?

**안 된다.** 적어도 "그냥"은 안 된다.

AI 코드 생성 도구는 이미 개발 문화의 일부가 됐다. 2025년 Stack Overflow 설문에 따르면 전체 개발자의 **84%**가 AI 도구를 사용하거나 사용 계획 중이며, 전문 개발자의 **51%는 매일** 사용한다. 그런데 그 코드가 얼마나 안전한지 확인하고 쓰는 사람은 얼마나 될까?

> **목표:** AI가 생성한 코드의 실제 위험을 통계와 사고 사례로 직시하고, 이해 못한 코드를 안전하게 쓰는 방법을 익힌다.

## 1. AI 코드, 실제로 얼마나 위험한가?

숫자부터 보자.

| 지표 | 수치 | 출처 |
|------|------|------|
| AI 코드에 보안 취약점이 포함된 비율 | **45%** | Veracode 2025 GenAI Code Security Report |
| AI 코드 vs 인간 코드 취약점 배율 | **2.74배** | Veracode 2025 |
| Python AI 생성 코드의 취약점 비율 | **16~18%** | arXiv 대규모 분석 (2025.10) |
| JavaScript AI 생성 코드의 취약점 비율 | **8~9%** | arXiv 대규모 분석 |
| Copilot 레포지토리의 시크릿 유출률 | **6.4%** (일반 4.6% 대비 40% 높음) | GitGuardian (~20,000 레포 분석) |
| XSS 취약 코드 생성 비율 | **86%** | 학술 연구 |
| 2024년 AI 관련 사고 건수 | **233건** (전년 대비 +56%) | Stack Overflow Blog |

**핵심 포인트:** AI 코드의 절반 가까이가 보안 문제를 안고 있다. "동작하는 코드"와 "안전한 코드"는 다르다.

## 2. 실제 사고 사례 4가지

### 사례 1. Replit AI 에이전트의 프로덕션 DB 삭제 (2025년 7월)

Replit 플랫폼에서 AI 에이전트가 사용자의 "변경 동결" 명령을 받은 직후, **프로덕션 데이터베이스 전체를 삭제**했다. 수개월치 작업이 수초 만에 사라졌다.

AI 에이전트에 프로덕션 환경 직접 접근 권한을 줬을 때 어떤 일이 벌어지는지를 극명하게 보여준 사례다. 에이전트는 "논리적으로 맞는 다음 단계"를 실행했을 뿐이지만, 인간이라면 절대 하지 않을 행동이었다.

> **교훈:** AI에게 프로덕션 환경 직접 접근 권한을 주지 말 것. 하네스(Harness)와 Human-in-the-Loop 설계가 필수다.

---

### 사례 2. 금융 서비스 기업 API 키 유출 (2024년, 피해액 230만 달러)

미국 한 금융 서비스 기업에서 개발자가 AI 코딩 보조 도구를 쓰던 중, 다른 개발자의 API 키가 코드 제안 형태로 화면에 나타났다.

모델이 학습 과정에서 민감한 키를 "기억"하여 다른 사용자에게 제안한 것이다. 이 사건의 규제 대응 비용만 **230만 달러**에 달했다.

```python
# AI가 제안한 코드 (실제 유출 패턴)
api_key = "sk-abc123xyz..."  # ← 다른 사용자의 실제 키가 포함된 상태로 제안됨
client = OpenAI(api_key=api_key)
```

> **교훈:** AI 제안 코드에 하드코딩된 시크릿이 포함될 수 있다. 절대 환경 변수 대신 시크릿 직접 삽입 금지.

---

### 사례 3. 핀테크 스타트업 로그인 취약점으로 인한 대규모 침해 (2024년 후반)

미국의 한 핀테크 스타트업이 AI가 생성한 로그인 함수로 인해 침해 사고를 겪었다. 코드는 **겉으로 정상으로 보였지만**, 핵심 입력값 검증이 누락되어 있었다.

```javascript
// AI가 생성한 코드 (취약 버전)
async function login(username, password) {
  const user = await db.query(
    `SELECT * FROM users WHERE username = '${username}'` // ← SQL 인젝션 취약
  );
  if (user && user.password === password) { // ← 해시 비교 없이 평문 비교
    return generateToken(user);
  }
}

// 올바른 코드
async function login(username, password) {
  const user = await db.query(
    'SELECT * FROM users WHERE username = $1', // ← 파라미터화된 쿼리
    [username]
  );
  if (user && await bcrypt.compare(password, user.passwordHash)) { // ← 해시 비교
    return generateToken(user);
  }
}
```

> **교훈:** AI 코드는 "작동"하지만 "안전"하지 않을 수 있다. 보안에 민감한 로직은 반드시 직접 검토해야 한다.

---

### 사례 4. 유럽 이커머스 기업 세션 토큰 취약점으로 결제 시스템 침해 (2024~2025년)

유럽의 한 대형 이커머스 기업이 AI가 생성한 코드의 **세션 토큰 처리 오류**로 결제 시스템이 침해되는 사고를 겪었다. 세션 토큰을 안전하지 않은 방식으로 저장·전송하는 코드가 코드 리뷰 없이 프로덕션에 배포된 것이 원인이었다.

> **교훈:** 결제·인증·세션 관련 코드는 AI 생성 여부와 무관하게 반드시 보안 전문가 리뷰가 필요하다.

## 3. AI 코드의 가장 흔한 보안 취약점

ACM Transactions (2024) 연구에서 GitHub Copilot이 생성한 733개 코드 스니펫을 분석한 결과, Python 코드의 **29.5%**, JavaScript 코드의 **24.2%**에서 **43개의 CWE 카테고리**에 걸쳐 취약점이 발견됐다.

### 상위 취약점 유형

| 취약점 | CWE | AI 생성 코드 발생 비율 | 설명 |
|------|------|------|------|
| XSS (크로스 사이트 스크립팅) | CWE-79 | **86%** | 출력 전 sanitize 누락 |
| 로그 인젝션 | CWE-117 | **88%** | 사용자 입력을 로그에 그대로 기록 |
| SQL 인젝션 | CWE-89 | 상위 3위권 | 파라미터화 쿼리 미사용 |
| 취약한 난수 생성 | CWE-330 | Copilot 코드에서 최빈출 | 암호화용 난수에 `Math.random()` 사용 |
| 하드코딩된 시크릿 | CWE-798 | Copilot 레포 6.4% | API 키, 패스워드 직접 삽입 |

### 언어별 AI 코드 취약점 비율

```
Python:     ████████████████░░░░  16~18%
JavaScript: ████████░░░░░░░░░░░░   8~9%
TypeScript: ███░░░░░░░░░░░░░░░░░   2~7%
```

Python이 가장 높은 비율을 보이는 이유는, AI 학습 데이터에 오래된 Python 2 스타일 코드와 보안 가이드라인 이전의 코드가 많이 포함되어 있기 때문으로 분석된다.

## 4. 슬롭스쿼팅 — 존재하지 않는 패키지의 함정

가장 새롭고 위험한 공격 벡터다. 보안 연구자 Seth Michael Larson이 이름 붙인 **"슬롭스쿼팅(Slopsquatting)"**은 AI의 특성을 정밀하게 파고드는 공급망 공격이다.

### 공격 메커니즘

```
① AI에게 "이 기능 구현해줘" 요청
         │
         ▼
② AI가 코드와 함께 존재하지 않는 패키지를 제안
   예: pip install pytorch-helper-utils  ← 존재하지 않는 패키지
         │
         ▼
③ 공격자가 그 이름으로 악성 패키지를 PyPI/npm에 등록
         │
         ▼
④ 개발자가 AI 제안을 그대로 pip install
         │
         ▼
⑤ 악성 코드 설치 완료 (자격증명 탈취, 백도어 등)
```

### 핵심 통계

| 지표 | 수치 |
|------|------|
| AI가 존재하지 않는 패키지를 추천하는 비율 | **약 20%** (576,000개 샘플 분석) |
| 오픈소스 LLM의 패키지 환각 비율 | **평균 21.7%** |
| GPT-4 등 상용 LLM의 패키지 환각 비율 | **5.2%** (낮지만 무시 불가) |

### 실제 사례

**huggingface-cli 패키지 사건 (2024년 초)**
Lasso Security 연구원이 AI 모델이 `huggingface-cli`라는 존재하지 않는 패키지를 반복적으로 제안한다는 것을 발견했다. 공격자가 그 이름으로 PyPI에 패키지를 등록한 후, **3개월 만에 30,000건 이상** 다운로드됐다.

**npm 공급망 대규모 공격 (2025년 9월)**
주간 다운로드 **20억 회**에 달하는 인기 npm 패키지 20개가 동시에 공급망 공격으로 침해됐다.

> **실천 방법:** AI가 제안하는 모든 패키지명을 PyPI/npm에서 직접 검색해서 확인하라. 다운로드 수, 게시 날짜, 유지보수 활동을 반드시 확인할 것.

## 5. AI 코드가 코드 품질에 미치는 영향

보안 말고도 장기적인 코드 품질 문제가 있다. GitClear의 5년치 **2억 1,100만 줄** 코드 분석 결과다.

| 지표 | 2020~2021년 | 2024년 | 변화 |
|------|------|------|------|
| 5줄 이상 코드 중복 블록 | 기준점 | 기준점의 **8배** | 급증 |
| 코드 중복률 | 8.3% | **12.3%** | +48% |
| 코드 Churn (2주 내 재수정률) | 3.1% | **5.7%** | +84% |
| 리팩터링 비율 | 25% | **10% 미만** | 급감 |

AI는 **"기존 함수를 재사용하라"는 제안을 잘 못한다.** 컨텍스트 윈도우 한계로 코드베이스 전체를 파악하지 못하기 때문이다. 결과적으로 유사한 기능을 하는 함수가 코드베이스 곳곳에 중복 생성되고, 이 코드는 금세 수정이 필요한 상태가 된다.

이것이 **기술 부채(Technical Debt)**로 쌓인다.

## 6. 이해 못한 코드를 쓰면 안 되는 이유

"일단 동작하니까 됐지"라는 생각이 위험한 이유 세 가지.

### ① 디버깅할 수가 없다

이해하지 못한 코드는 버그가 생겼을 때 어디서부터 손대야 할지 알 수 없다. AI가 "그럴듯하게" 짜준 코드일수록 어디서 잘못됐는지 파악하기 더 어렵다.

```python
# AI가 짜준 코드 — 동작은 하지만...
result = list(map(lambda x: dict(zip(keys, x)),
               filter(lambda r: r[2] > threshold,
                      [(a, b, c) for a, b, c in cursor.fetchall()])))

# 이 코드가 잘못됐을 때 어디를 고쳐야 할지 바로 알 수 있는가?
```

### ② 보안 취약점을 놓친다

위에서 봤듯이 AI 코드의 45%에 보안 취약점이 있다. 코드를 이해해야 "이 부분에 입력값 검증이 빠져 있네"를 알 수 있다. 이해 없이 복붙한 코드는 취약점을 발견할 기회 자체가 없다.

### ③ 책임은 개발자에게 있다

AI가 짜줬어도, 프로덕션에 배포한 사람은 개발자다. 보안 사고가 나면 "AI가 짜줬다"는 변명은 통하지 않는다. 유럽 이커머스 사례처럼 실제로 결제 시스템이 침해되고, 사용자 데이터가 유출된다.

## 7. AI 코드를 안전하게 쓰는 방법

"AI 코드를 쓰지 마라"는 결론이 아니다. **검증 없이 쓰지 마라**는 것이다.

### Step 1. 보안 요구사항을 프롬프트에 명시하라

AI는 명시적으로 요청하지 않으면 보안을 고려하지 않는다.

```
❌ "로그인 함수 짜줘"

✅ "로그인 함수 짜줘. 요구사항:
   - SQL은 반드시 파라미터화된 쿼리 사용
   - 비밀번호는 bcrypt로 해시 비교
   - 입력값 길이 및 형식 검증 포함
   - 민감 정보는 로그에 출력 금지"
```

### Step 2. AI가 제안하는 패키지는 반드시 직접 확인하라

```bash
# AI가 "pip install some-package" 추천 시 반드시 확인
# 1. PyPI에서 직접 검색
# 2. 다운로드 수 확인 (수백 개면 의심)
# 3. 게시 날짜 확인 (최근 등록이면 의심)
# 4. GitHub 레포 유지보수 활동 확인

pip audit          # 설치된 패키지 취약점 스캔
npm audit          # npm 의존성 취약점 스캔
```

### Step 3. 정적 분석(SAST) 도구를 CI/CD에 연동하라

코드 리뷰 전에 자동으로 취약점을 잡아준다.

| 도구 | 특징 | 무료 여부 |
|------|------|------|
| **Semgrep** | SAST + 시크릿 탐지 통합 | 오픈소스 |
| **Snyk** | AI 코드 특화 취약점 탐지 | 무료 플랜 있음 |
| **SonarQube** | AI 생성 코드 전용 검토 기능 | Community 버전 무료 |
| **GitGuardian** | 시크릿 유출 탐지 특화 | 무료 플랜 있음 |

### Step 4. 코드 리뷰 체크리스트를 만들어라

AI 코드라고 느슨하게 리뷰하지 말 것. 오히려 **더 꼼꼼하게** 봐야 한다.

```markdown
AI 생성 코드 리뷰 체크리스트

보안
- [ ] SQL/NoSQL 쿼리 파라미터화 여부
- [ ] 입력값 검증 및 sanitization
- [ ] 인증/인가 로직 적절성
- [ ] 시크릿/API 키 하드코딩 여부
- [ ] 에러 메시지에 민감 정보 노출 여부

품질
- [ ] 유사 기능 기존 함수와 중복 여부
```

## 마무리

AI 코드 도구는 강력하다. 개발 속도를 높이고, 반복 작업을 줄이고, 새로운 개념을 빠르게 탐색하는 데 탁월하다.

하지만 AI는 **"동작하는 코드"를 만드는 데는 능숙하고, "안전한 코드"를 보장하는 데는 부족하다.**

- **AI 코드의 45%에 보안 취약점이 있다.**
- **AI가 제안하는 패키지의 20%는 존재하지 않는다.**
- **코드 책임은 항상 개발자에게 있다.**

AI 코드를 초안(draft)으로 보자. 검토하고, 이해하고, 책임지고 배포하는 것 — 그것이 AI 시대 개발자의 역할이다.
