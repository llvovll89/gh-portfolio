## 1. 자료구조

> 자료구조를 모르면 올바른 도구를 선택할 수 없다.

### 핵심 자료구조 비교

| 자료구조 | 접근 | 검색 | 삽입 | 삭제 | 언제 쓰나 |
|---|---|---|---|---|---|
| Array | O(1) | O(n) | O(n) | O(n) | 인덱스 기반 접근이 많을 때 |
| Linked List | O(n) | O(n) | O(1) | O(1) | 삽입/삭제가 빈번할 때 |
| Hash Map | O(1) | O(1) | O(1) | O(1) | 키-값 조회가 많을 때 |
| Binary Search Tree | O(log n) | O(log n) | O(log n) | O(log n) | 정렬된 데이터 + 범위 조회 |
| Stack | O(n) | O(n) | O(1) | O(1) | LIFO, 함수 콜스택, undo |
| Queue | O(n) | O(n) | O(1) | O(1) | FIFO, BFS, 작업 큐 |

### 스택 vs 큐 — 직접 구현해보기

```typescript
// Stack — Last In, First Out
class Stack<T> {
  private items: T[] = [];

  push(item: T) { this.items.push(item); }
  pop(): T | undefined { return this.items.pop(); }
  peek(): T | undefined { return this.items[this.items.length - 1]; }
  isEmpty(): boolean { return this.items.length === 0; }
}

// Queue — First In, First Out
class Queue<T> {
  private items: T[] = [];

  enqueue(item: T) { this.items.push(item); }
  dequeue(): T | undefined { return this.items.shift(); }
  isEmpty(): boolean { return this.items.length === 0; }
}
```

### 해시 충돌 해결 방법

- **체이닝(Chaining):** 같은 버킷에 연결 리스트로 저장 → 최악 O(n)
- **오픈 어드레싱(Open Addressing):** 빈 버킷을 찾아 이동 → 캐시 친화적
- **로드 팩터(Load Factor):** 보통 0.75 초과 시 리해싱 수행

## 2. 알고리즘

### 시간 복잡도 감각 익히기

```
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!)
```

n = 1,000,000 기준:
- `O(log n)` → 약 20회 연산 ← 이진 탐색
- `O(n log n)` → 약 20,000,000회 ← 정렬 알고리즘의 이상적인 목표
- `O(n²)` → 1,000,000,000,000회 ← 실무에서 절대 피해야 할 상황

### 정렬 알고리즘 비교

| 알고리즘 | 평균 | 최악 | 공간 | 안정성 |
|---|---|---|---|---|
| Bubble Sort | O(n²) | O(n²) | O(1) | 안정 |
| Quick Sort | O(n log n) | O(n²) | O(log n) | 불안정 |
| Merge Sort | O(n log n) | O(n log n) | O(n) | 안정 |
| Heap Sort | O(n log n) | O(n log n) | O(1) | 불안정 |
| Tim Sort | O(n log n) | O(n log n) | O(n) | 안정 |

> **실무 팁:** JavaScript의 `Array.prototype.sort()`는 V8에서 Tim Sort를 사용합니다.

### 이진 탐색 — 코드로 이해하기

```typescript
function binarySearch(arr: number[], target: number): number {
  let left = 0, right = arr.length - 1;

  while (left <= right) {
    const mid = Math.floor((left + right) / 2);
    if (arr[mid] === target) return mid;
    if (arr[mid] < target) left = mid + 1;
    else right = mid - 1;
  }

  return -1; // 없음
}
```

### 동적 프로그래밍 (DP) 사고법

DP는 **반복되는 하위 문제**를 메모이제이션으로 해결하는 기법입니다.

```
문제 → 점화식 세우기 → 베이스 케이스 정의 → 구현
```

```typescript
// 피보나치 — 메모이제이션
function fib(n: number, memo: Map<number, number> = new Map()): number {
  if (n <= 1) return n;
  if (memo.has(n)) return memo.get(n)!;
  const result = fib(n - 1, memo) + fib(n - 2, memo);
  memo.set(n, result);
  return result;
}
// O(2ⁿ) → O(n) 으로 개선
```

## 3. 운영체제

### 프로세스 vs 스레드

| 항목 | 프로세스 | 스레드 |
|---|---|---|
| 메모리 | 독립된 주소 공간 | 공유 주소 공간 |
| 통신 비용 | IPC 필요 (느림) | 공유 메모리 (빠름) |
| 격리성 | 강함 (크래시 영향 없음) | 약함 (한 스레드 크래시 → 전체 위험) |
| 생성 비용 | 높음 | 낮음 |

### 교착상태(Deadlock) — 4가지 조건

교착상태는 다음 **4가지 조건이 동시에 충족**될 때 발생합니다:

1. **상호 배제 (Mutual Exclusion):** 자원을 하나만 점유 가능
2. **점유 대기 (Hold and Wait):** 자원을 가진 채 다른 자원을 기다림
3. **비선점 (No Preemption):** 강제로 자원을 빼앗을 수 없음
4. **순환 대기 (Circular Wait):** A→B→C→A 형태의 순환 대기

> **예방법:** 자원 획득 순서를 전역적으로 고정하면 순환 대기가 발생하지 않습니다.

### 메모리 관리 — 가상 메모리와 페이징

```
가상 주소 공간
  └─ 페이지 테이블
       └─ 물리 메모리 (RAM) 또는 스왑 영역 (디스크)
```

- **페이지 폴트:** 필요한 페이지가 RAM에 없을 때 디스크에서 로드 → 성능 저하
- **스래싱(Thrashing):** 페이지 교체가 너무 빈번해 CPU가 I/O 대기에만 소비되는 상태

### CPU 스케줄링 알고리즘

| 알고리즘 | 특징 | 단점 |
|---|---|---|
| FCFS | 도착 순서대로 처리 | 호위 효과 (긴 작업이 짧은 작업을 블록) |
| SJF | 가장 짧은 작업 먼저 | 기아(Starvation) 문제 |
| Round Robin | 시간 할당량(퀀텀)으로 교대 | 퀀텀 크기 선택이 중요 |
| Priority | 우선순위 기반 | 기아 문제 (에이징으로 해결) |

## 4. 네트워크

### OSI 7계층 → TCP/IP 4계층

```
OSI 7계층              TCP/IP 4계층
─────────────          ─────────────
응용 (Application)  ┐
표현 (Presentation) ├→ 응용 (Application)   HTTP, DNS, SMTP
세션 (Session)      ┘
전송 (Transport)    →  전송 (Transport)     TCP, UDP
네트워크 (Network)  →  인터넷 (Internet)    IP, ICMP
데이터링크 (Data)   ┐
물리 (Physical)     ┘→ 네트워크 (Network)   Ethernet, Wi-Fi
```

### TCP vs UDP

| 항목 | TCP | UDP |
|---|---|---|
| 연결 | 연결 지향 (3-way handshake) | 비연결 |
| 신뢰성 | 순서 보장, 재전송 | 보장 없음 |
| 속도 | 느림 | 빠름 |
| 사용처 | HTTP, SSH, 파일 전송 | DNS, 게임, 스트리밍 |

### HTTP/1.1 vs HTTP/2 vs HTTP/3

- **HTTP/1.1:** 요청마다 연결, Head-of-Line Blocking 문제
- **HTTP/2:** 멀티플렉싱 (한 연결에 여러 스트림), 헤더 압축 (HPACK)
- **HTTP/3:** QUIC 기반 (UDP), 연결 재설정 시간 단축, 0-RTT

### HTTPS와 TLS 핸드셰이크

```
클라이언트            서버
    │── ClientHello ──→│  (지원하는 암호 스위트 목록)
    │← ServerHello ───│  (선택된 암호 스위트 + 인증서)
    │── 키 교환 ───────│  (공개키로 암호화)
    │← 완료 ──────────│
    │═══ 암호화 통신 ═══│
```

## 5. 데이터베이스

### ACID 속성

| 속성 | 설명 |
|---|---|
| **A**tomicity (원자성) | 트랜잭션은 전부 성공하거나 전부 실패 |
| **C**onsistency (일관성) | 트랜잭션 전후 데이터 무결성 유지 |
| **I**solation (격리성) | 동시 트랜잭션이 서로 영향을 주지 않음 |
| **D**urability (영속성) | 커밋된 데이터는 시스템 장애에도 보존 |

### 인덱스 — 왜 중요한가

```sql
-- 인덱스 없음: Full Table Scan → O(n)
SELECT * FROM users WHERE email = 'test@example.com';

-- 인덱스 있음: B-Tree 탐색 → O(log n)
CREATE INDEX idx_users_email ON users(email);
```

**인덱스의 단점:**
- 쓰기(INSERT/UPDATE/DELETE) 성능 저하
- 추가 디스크 공간 소비
- 과도한 인덱스 = 오히려 옵티마이저 혼란

### 정규화 — 1NF / 2NF / 3NF

- **1NF:** 원자값만 저장 (다중 값 컬럼 금지)
- **2NF:** 부분 함수 종속 제거 (복합 기본키의 일부에만 의존하는 컬럼 분리)
- **3NF:** 이행 함수 종속 제거 (비키 컬럼이 다른 비키 컬럼에 의존하는 구조 분리)

### SQL vs NoSQL

| 항목 | SQL (RDB) | NoSQL |
|---|---|---|
| 스키마 | 고정 | 유연 |
| 조인 | 강력 | 제한적 |
| 확장성 | 수직 확장 | 수평 확장 |
| 일관성 | 강한 일관성 | 최종 일관성 |
| 사용처 | 금융, ERP | SNS, 로그, 캐시 |

## 6. 실전 학습 전략

```

3단계 (실전 적용)
```
