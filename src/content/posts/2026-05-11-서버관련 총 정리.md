서버(Server)는 클라이언트의 요청을 받아 처리하고 응답을 반환하는 컴퓨터 혹은 프로그램입니다. 웹 개발에서 "서버를 안다"는 것은 단순히 백엔드 코드를 작성하는 수준을 넘어, 네트워크 통신 원리·인프라 구성·배포 파이프라인까지 아우르는 넓은 영역을 의미합니다. 이 글에서는 프론트엔드 개발자도 반드시 알아야 할 서버 지식을 체계적으로 정리합니다.

> 목표: HTTP 통신 원리부터 Nginx 설정, Node.js 서버 구성, 클라우드 서비스, CI/CD 배포 전략까지 실무에서 바로 쓸 수 있는 서버 지식을 한 번에 습득한다.

## 1. 서버의 기본 개념

### 클라이언트-서버 모델

```
클라이언트 (브라우저)          서버
      │                          │
      │──── HTTP Request ────────▶│
      │                          │ 처리 (DB 조회, 로직 실행)
      │◀─── HTTP Response ────────│
      │                          │
```

- **클라이언트**: 서비스를 요청하는 측 (브라우저, 모바일 앱, CLI 등)
- **서버**: 요청을 받아 처리하고 응답을 반환하는 측
- **프로토콜**: 통신 규약 (HTTP, HTTPS, WebSocket, gRPC 등)

### 서버의 종류

| 종류 | 역할 | 예시 |
|------|------|------|
| 웹 서버 | 정적 파일 제공, 리버스 프록시 | Nginx, Apache |
| WAS | 동적 콘텐츠 처리, 비즈니스 로직 | Node.js, Spring, Django |
| DB 서버 | 데이터 저장 및 조회 | PostgreSQL, MySQL, MongoDB |
| 파일 서버 | 파일 저장 및 전달 | S3, MinIO |
| 캐시 서버 | 빠른 데이터 임시 저장 | Redis, Memcached |
| 메시지 브로커 | 비동기 메시지 큐 | RabbitMQ, Kafka |

## 2. HTTP 프로토콜 심화

### HTTP 메서드

| 메서드 | 의미 | 멱등성 | 안전성 |
|--------|------|--------|--------|
| GET | 리소스 조회 | ✅ | ✅ |
| POST | 리소스 생성 | ❌ | ❌ |
| PUT | 리소스 전체 교체 | ✅ | ❌ |
| PATCH | 리소스 부분 수정 | ❌ | ❌ |
| DELETE | 리소스 삭제 | ✅ | ❌ |
| OPTIONS | 허용 메서드 조회 | ✅ | ✅ |

> **멱등성(Idempotent)**: 같은 요청을 여러 번 해도 결과가 동일  
> **안전성(Safe)**: 서버 상태를 변경하지 않음

### HTTP 상태 코드

```
1xx  정보성     100 Continue, 101 Switching Protocols
2xx  성공       200 OK, 201 Created, 204 No Content
3xx  리다이렉션 301 Moved Permanently, 302 Found, 304 Not Modified
4xx  클라이언트 오류  400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests
5xx  서버 오류  500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable
```

### HTTP 헤더 핵심

```http
# 요청 헤더
GET /api/users HTTP/1.1
Host: example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Content-Type: application/json
Accept: application/json
Cache-Control: no-cache

# 응답 헤더
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Cache-Control: max-age=3600
Access-Control-Allow-Origin: https://example.com
ETag: "abc123"
```

### HTTP/1.1 vs HTTP/2 vs HTTP/3

| 버전 | 특징 | 연결 방식 |
|------|------|-----------|
| HTTP/1.1 | 요청당 1 TCP 연결, Keep-Alive | TCP |
| HTTP/2 | 멀티플렉싱, 헤더 압축, 서버 푸시 | TCP |
| HTTP/3 | QUIC 기반, 0-RTT, 패킷 손실 개선 | UDP(QUIC) |

### CORS (Cross-Origin Resource Sharing)

```js
// Express에서 CORS 설정
import cors from 'cors';

app.use(cors({
  origin: ['https://example.com', 'http://localhost:3000'],
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  credentials: true,  // 쿠키 허용
  maxAge: 86400,      // preflight 캐시 시간(초)
}));
```

**Preflight 요청**: `Content-Type: application/json` 등 "simple request"가 아닌 경우 브라우저가 먼저 `OPTIONS` 요청을 보내 서버 허용 여부를 확인합니다.

## 3. 웹 서버 vs WAS

```
인터넷
  │
  ▼
[Nginx] ──────── 정적 파일 (.html, .css, .js, 이미지)
  │
  │ 동적 요청 프록시
  ▼
[Node.js / Spring / Django]  ──── 비즈니스 로직
  │
  ▼
[Database / Cache / 외부 API]
```

### 역할 분리의 이유

- **성능**: Nginx는 정적 파일을 C로 작성된 이벤트 루프로 처리 → 매우 빠름
- **보안**: WAS를 직접 노출하지 않고 Nginx가 앞단에서 필터링
- **확장성**: Nginx ↔ WAS 사이에 로드 밸런서 배치 가능
- **SSL 종료**: Nginx에서 HTTPS 처리 후 내부는 HTTP로 통신

## 4. Nginx 핵심 설정

### 기본 구조

```nginx
# /etc/nginx/nginx.conf
events {
  worker_connections 1024;
}

http {
  include       mime.types;
  default_type  application/octet-stream;
  sendfile      on;
  keepalive_timeout 65;

  include /etc/nginx/conf.d/*.conf;
}
```

### 사이트 설정 파일

```nginx
# /etc/nginx/conf.d/myapp.conf
server {
  listen 80;
  server_name example.com www.example.com;

  # HTTP → HTTPS 리다이렉트
  return 301 https://$host$request_uri;
}

server {
  listen 443 ssl http2;
  server_name example.com;

  ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

  # 정적 파일 서빙
  root /var/www/myapp/dist;
  index index.html;

  # SPA 라우팅 (React Router 등)
  location / {
    try_files $uri $uri/ /index.html;
  }

  # API 리버스 프록시
  location /api/ {
    proxy_pass         http://localhost:4000;
    proxy_http_version 1.1;
    proxy_set_header   Upgrade $http_upgrade;
    proxy_set_header   Connection 'upgrade';
    proxy_set_header   Host $host;
    proxy_cache_bypass $http_upgrade;
    proxy_set_header   X-Real-IP $remote_addr;
    proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
  }

  # gzip 압축
  gzip on;
  gzip_types text/plain text/css application/json application/javascript;
  gzip_min_length 1024;
}
```

### 로드 밸런싱

```nginx
upstream backend {
  least_conn;  # 최소 연결 알고리즘
  server 127.0.0.1:4001 weight=3;
  server 127.0.0.1:4002 weight=2;
  server 127.0.0.1:4003 backup;  # 나머지 모두 다운 시 사용
}

server {
  location /api/ {
    proxy_pass http://backend;
  }
}
```

## 5. Node.js 서버 구성

### Express 기본 구조

```ts
// src/app.ts
import express from 'express';
import cors from 'cors';
import helmet from 'helmet';
import morgan from 'morgan';
import { userRouter } from './routes/user.js';
import { errorHandler } from './middleware/errorHandler.js';

const app = express();

// 미들웨어 체인
app.use(helmet());                    // 보안 헤더
app.use(cors({ origin: process.env.CLIENT_URL }));
app.use(morgan('combined'));          // 요청 로깅
app.use(express.json({ limit: '10mb' }));
app.use(express.urlencoded({ extended: true }));

// 라우트
app.use('/api/users', userRouter);

// 전역 에러 핸들러 (마지막에 등록)
app.use(errorHandler);

export default app;
```

### 미들웨어 패턴

```ts
// 인증 미들웨어
import { Request, Response, NextFunction } from 'express';
import jwt from 'jsonwebtoken';

export const authenticate = (req: Request, res: Response, next: NextFunction) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ message: '토큰 없음' });

  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET!);
    req.user = payload as JwtPayload;
    next();
  } catch {
    res.status(401).json({ message: '유효하지 않은 토큰' });
  }
};
```

### 에러 핸들링

```ts
// middleware/errorHandler.ts
export class AppError extends Error {
  constructor(
    public statusCode: number,
    message: string,
  ) {
    super(message);
    this.name = 'AppError';
  }
}

export const errorHandler = (
  err: Error,
  _req: Request,
  res: Response,
  _next: NextFunction,
) => {
  if (err instanceof AppError) {
    return res.status(err.statusCode).json({ message: err.message });
  }
  console.error(err);
  res.status(500).json({ message: '서버 내부 오류' });
};
```

## 7. 인증과 보안

### JWT (JSON Web Token)

```
Header.Payload.Signature

eyJhbGciOiJIUzI1NiJ9  .  eyJ1c2VySWQiOiIxMjMifQ  .  SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
     헤더 (Base64)              페이로드 (Base64)              서명 (HMAC-SHA256)
```

```ts
// 토큰 발급
const accessToken = jwt.sign(
  { userId: user.id, role: user.role },
  process.env.JWT_SECRET!,
  { expiresIn: '15m' }           // 단기 토큰
);

const refreshToken = jwt.sign(
  { userId: user.id },
  process.env.REFRESH_SECRET!,
  { expiresIn: '7d' }            // 장기 토큰, DB에 저장
);
```

### 세션 vs JWT 비교

| 항목 | 세션 | JWT |
|------|------|-----|
| 저장 위치 | 서버 (DB/Redis) | 클라이언트 |
| 서버 상태 | Stateful | Stateless |
| 확장성 | 분산 환경 복잡 | 수평 확장 용이 |
| 무효화 | 즉시 가능 | 만료까지 대기 |
| 크기 | 작음 (세션 ID) | 큼 (토큰 자체) |

### 보안 체크리스트

- [ ] **HTTPS** 적용 (Let's Encrypt 무료 인증서)
- [ ] **환경 변수** 분리 (`.env` 파일, 코드에 시크릿 하드코딩 금지)
- [ ] **SQL Injection** 방지 — Prepared Statement, ORM 사용
- [ ] **XSS** 방지 — 입력값 검증, `helmet` 미들웨어
- [ ] **CSRF** 방지 — SameSite 쿠키, CSRF 토큰
- [ ] **Rate Limiting** — `express-rate-limit`으로 무차별 대입 공격 방지
- [ ] **bcrypt** 비밀번호 해싱 (saltRounds ≥ 10)
- [ ] **의존성 취약점** 정기 점검 — `npm audit`

## 8. 데이터베이스 연동

### SQL vs NoSQL 선택 기준

| 기준 | SQL (PostgreSQL, MySQL) | NoSQL (MongoDB, Firestore) |
|------|------------------------|---------------------------|
| 스키마 | 고정 스키마 | 유연한 스키마 |
| 관계 | JOIN으로 복잡한 관계 처리 | 참조 또는 임베딩 |
| 트랜잭션 | ACID 완전 지원 | 제한적 |
| 확장 | 수직 확장 중심 | 수평 확장 용이 |
| 적합 사례 | 금융, ERP, 복잡한 쿼리 | 실시간, 유연한 데이터 구조 |

### Connection Pool

```ts
// PostgreSQL with pg
import { Pool } from 'pg';

const pool = new Pool({
  host: process.env.DB_HOST,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  database: process.env.DB_NAME,
  max: 20,          // 최대 연결 수
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
});

// 쿼리 실행
const { rows } = await pool.query(
  'SELECT * FROM users WHERE id = $1',
  [userId]         // Prepared Statement (SQL Injection 방지)
);
```

### Redis 캐싱 패턴

```ts
// Cache-Aside 패턴
async function getUser(id: string) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);

  const user = await db.findUser(id);
  await redis.setex(`user:${id}`, 3600, JSON.stringify(user)); // TTL 1시간
  return user;
}
```

## 10. 배포 전략과 CI/CD

### 배포 전략

```
블루-그린(Blue-Green) 배포
  [로드 밸런서]
       │
  ┌────┴────┐
[Blue v1.0] [Green v1.1]  ← 새 버전 배포 후 트래픽 전환
  (현재 운영)  (대기)

롤링(Rolling) 배포
  인스턴스를 하나씩 새 버전으로 교체 (서비스 중단 없음)

카나리(Canary) 배포
  트래픽 5% → 새 버전으로 → 문제 없으면 점진적 확대
```

### GitHub Actions CI/CD

```yaml
# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run build

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
```

### Docker 기본

## 11. 성능 최적화

### 서버 사이드 캐싱 전략

```
요청 → Nginx 캐시 → Redis 캐시 → 애플리케이션 → DB

1. Nginx 캐시: 완성된 응답을 파일로 저장 (정적 콘텐츠에 효과적)
2. Redis 캐시: 자주 조회하는 DB 결과를 메모리에 보관
3. DB 쿼리 캐시: ORM 레벨의 쿼리 결과 캐싱
```

### HTTP 캐싱 헤더

```ts
// 정적 자산 — 1년 캐시 (파일명에 해시 포함 시)
res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');

// API 응답 — 캐시 금지
res.setHeader('Cache-Control', 'no-store');

// 조건부 캐싱 (304 Not Modified 활용)
res.setHeader('ETag', etag(data));
res.setHeader('Cache-Control', 'public, max-age=0, must-revalidate');
```

### N+1 문제 해결

```ts
// ❌ N+1: 게시글 N개마다 작성자 쿼리 N번 추가 발생
const posts = await Post.findAll();
for (const post of posts) {
  post.author = await User.findByPk(post.userId); // N번 쿼리
}

// ✅ 해결: JOIN 또는 include
const posts = await Post.findAll({
  include: [{ model: User, as: 'author' }], // 1번 쿼리
});
```

### 로깅 전략 (Winston)

```ts
import winston from 'winston';

const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.errors({ stack: true }),
    winston.format.json(),
  ),
  transports: [
    new winston.transports.Console(),
    new winston.transports.File({ filename: 'logs/error.log', level: 'error' }),
    new winston.transports.File({ filename: 'logs/combined.log' }),
  ],
});

// 사용
logger.info('서버 시작', { port: 4000 });
logger.error('DB 연결 실패', { error: err.message });
```

### 헬스체크 엔드포인트

```ts
app.get('/health', async (req, res) => {
  const checks = {
    status: 'ok',
    timestamp: new Date().toISOString(),
    uptime: process.uptime(),
    db: await checkDatabase(),
    redis: await checkRedis(),

  const status = Object.values(checks).every(v => v !== 'error') ? 200 : 503;
  res.status(status).json(checks);
});
```

## 마무리

서버 지식은 한 번에 모든 것을 이해하기보다는 **필요한 부분을 깊이 파고드는 방식**으로 습득하는 게 효과적입니다. 프론트엔드 개발자라면 최소한 아래 세 가지는 확실히 이해해 두는 것을 권장합니다.

1. **HTTP 프로토콜** — 상태 코드, 헤더, CORS는 거의 매일 마주칩니다.
2. **배포 파이프라인** — CI/CD와 환경 변수 관리는 실무 필수 역량입니다.
3. **인증 흐름** — JWT, 세션 중 하나를 선택해 깊이 이해하세요.

나머지는 "필요할 때 찾아보는 것"이 아니라 "왜 이런 구조인지 이해한 후 필요할 때 구현"하는 방향으로 점진적으로 경험을 쌓아 나가시기 바랍니다.
