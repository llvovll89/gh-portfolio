## 1. 소프트웨어 아키텍처란?

소프트웨어 아키텍처는 시스템의 **구조적 뼈대**입니다. 어떤 컴포넌트들이 존재하고, 그것들이 어떻게 상호작용하며, 어떤 원칙에 따라 설계되는지를 정의합니다.

### 좋은 아키텍처의 특징

| 특징 | 설명 |
|------|------|
| **유지보수성** | 변경이 쉽고, 변경의 영향 범위가 제한적 |
| **확장성** | 새로운 기능 추가 시 기존 코드를 최소한으로 수정 |
| **테스트 용이성** | 각 컴포넌트를 독립적으로 테스트 가능 |
| **재사용성** | 컴포넌트를 다른 컨텍스트에서도 활용 가능 |
| **가독성** | 의도가 명확하게 드러나는 코드 구조 |

### 아키텍처 vs 디자인 패턴

```
아키텍처 (Architecture)
  └─ 시스템 전체의 큰 그림, 컴포넌트 간의 관계
      예: MVC, Clean Architecture, Hexagonal Architecture

디자인 패턴 (Design Pattern)
  └─ 특정 문제를 해결하는 재사용 가능한 설계 솔루션
      예: Singleton, Observer, Factory
```

## 2. SOLID 원칙

SOLID는 객체지향 설계의 5가지 핵심 원칙으로, 모든 디자인 패턴의 기반이 됩니다.

### S — 단일 책임 원칙 (Single Responsibility Principle)

> 클래스는 변경될 이유가 오직 하나여야 한다.

```typescript
// ❌ 나쁜 예: 한 클래스가 너무 많은 책임
class UserService {
  saveUser(user: User) { /* DB 저장 */ }
  sendWelcomeEmail(user: User) { /* 이메일 발송 */ }
  generateReport(users: User[]) { /* 리포트 생성 */ }
}

// ✅ 좋은 예: 책임 분리
class UserRepository {
  save(user: User) { /* DB 저장만 */ }
}

class EmailService {
  sendWelcomeEmail(user: User) { /* 이메일만 */ }
}

class UserReportGenerator {
  generate(users: User[]) { /* 리포트만 */ }
}
```

### O — 개방-폐쇄 원칙 (Open-Closed Principle)

> 소프트웨어 엔티티는 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.

```typescript
// ❌ 나쁜 예: 새 할인 정책 추가 시 기존 코드 수정 필요
function calculateDiscount(type: string, price: number) {
  if (type === 'vip') return price * 0.8;
  if (type === 'member') return price * 0.9;
  // 새 타입 추가 시 여기를 계속 수정해야 함
}

// ✅ 좋은 예: 인터페이스로 확장
interface DiscountPolicy {
  apply(price: number): number;
}

class VipDiscount implements DiscountPolicy {
  apply(price: number) { return price * 0.8; }
}

class MemberDiscount implements DiscountPolicy {
  apply(price: number) { return price * 0.9; }
}

// 새 정책은 클래스만 추가, 기존 코드 변경 없음
class SeasonalDiscount implements DiscountPolicy {
  apply(price: number) { return price * 0.7; }
}
```

### L — 리스코프 치환 원칙 (Liskov Substitution Principle)

> 자식 클래스는 부모 클래스를 완전히 대체할 수 있어야 한다.

```typescript
class Rectangle {
  constructor(protected width: number, protected height: number) {}
  area() { return this.width * this.height; }
  setWidth(w: number) { this.width = w; }
  setHeight(h: number) { this.height = h; }
}

// ❌ Square는 Rectangle의 행동 계약을 위반
class Square extends Rectangle {
  setWidth(w: number) {
    this.width = w;
    this.height = w; // 부모의 기대 동작과 다름
  }
}

// ✅ 공통 인터페이스로 해결
interface Shape {
  area(): number;
}

class Rectangle implements Shape { /* ... */ }
class Square implements Shape { /* ... */ }
```

### I — 인터페이스 분리 원칙 (Interface Segregation Principle)

> 클라이언트는 자신이 사용하지 않는 인터페이스에 의존하면 안 된다.

```typescript
// ❌ 너무 큰 인터페이스
interface Animal {
  eat(): void;
  fly(): void;  // 모든 동물이 날 수 있는 건 아님
  swim(): void; // 모든 동물이 수영하는 건 아님
}

// ✅ 분리된 인터페이스
interface Eatable { eat(): void; }
interface Flyable { fly(): void; }
interface Swimmable { swim(): void; }

class Duck implements Eatable, Flyable, Swimmable {
  eat() { /* ... */ }
  fly() { /* ... */ }
  swim() { /* ... */ }
}

class Dog implements Eatable, Swimmable {
  eat() { /* ... */ }
  swim() { /* ... */ }
}
```

### D — 의존성 역전 원칙 (Dependency Inversion Principle)

> 고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다.

```typescript
// ❌ 고수준 모듈이 저수준에 직접 의존
class OrderService {
  private db = new MySQLDatabase(); // 구체적 구현에 의존

  saveOrder(order: Order) {
    this.db.save(order);
  }
}

// ✅ 추상화(인터페이스)에 의존
interface Database {
  save(data: unknown): void;
}

class OrderService {
  constructor(private db: Database) {} // 인터페이스에 의존

  saveOrder(order: Order) {
    this.db.save(order);
  }
}

// DB 구현체를 자유롭게 교체 가능
const service = new OrderService(new MySQLDatabase());
const testService = new OrderService(new InMemoryDatabase());
```

## 3. GoF 디자인 패턴 개요

GoF(Gang of Four)는 1994년 Erich Gamma 외 3인이 집필한 "Design Patterns: Elements of Reusable Object-Oriented Software"에서 정의한 23개의 패턴입니다.

```
GoF 23가지 패턴
├── 생성 패턴 (5개): 객체 생성 방식
│   ├── Singleton
│   ├── Factory Method
│   ├── Abstract Factory
│   ├── Builder
│   └── Prototype
│
├── 구조 패턴 (7개): 클래스/객체 조합
│   ├── Adapter
│   ├── Bridge
│   ├── Composite
│   ├── Decorator
│   ├── Facade
│   ├── Flyweight
│   └── Proxy
│
└── 행동 패턴 (11개): 객체 간 상호작용
    ├── Chain of Responsibility
    ├── Command
    ├── Iterator
    ├── Mediator
    ├── Memento
    ├── Observer
    ├── State
    ├── Strategy
    ├── Template Method
    ├── Visitor
    └── Interpreter
```

## 4. 생성 패턴

### Singleton — 오직 하나의 인스턴스

전체 시스템에서 하나의 인스턴스만 존재해야 할 때 사용합니다.

```typescript
class ConfigManager {
  private static instance: ConfigManager | null = null;
  private config: Record<string, string> = {};

  private constructor() {
    // 외부에서 new 불가
  }

  static getInstance(): ConfigManager {
    if (!ConfigManager.instance) {
      ConfigManager.instance = new ConfigManager();
    }
    return ConfigManager.instance;
  }

  get(key: string): string {
    return this.config[key];
  }

  set(key: string, value: string): void {
    this.config[key] = value;
  }
}

// 사용
const config = ConfigManager.getInstance();
config.set('apiUrl', 'https://api.example.com');

// 어디서 접근해도 같은 인스턴스
const sameConfig = ConfigManager.getInstance();
console.log(sameConfig.get('apiUrl')); // 'https://api.example.com'
```

> **주의:** Singleton은 전역 상태를 만들어 테스트를 어렵게 할 수 있습니다. 꼭 필요한 경우에만 사용하세요.

### Factory Method — 생성 로직 캡슐화

```typescript
interface Button {
  render(): void;
  onClick(handler: () => void): void;
}

class WindowsButton implements Button {
  render() { console.log('Windows 스타일 버튼 렌더링'); }
  onClick(handler: () => void) { /* Windows 이벤트 */ }
}

class MacButton implements Button {
  render() { console.log('Mac 스타일 버튼 렌더링'); }
  onClick(handler: () => void) { /* Mac 이벤트 */ }
}

// Factory: 어떤 버튼을 만들지 결정
abstract class Dialog {
  abstract createButton(): Button; // Factory Method

  render() {
    const button = this.createButton();
    button.render();
  }
}

class WindowsDialog extends Dialog {
  createButton(): Button { return new WindowsButton(); }
}

class MacDialog extends Dialog {
  createButton(): Button { return new MacButton(); }
}
```

### Builder — 복잡한 객체를 단계적으로

```typescript
class QueryBuilder {
  private table = '';
  private conditions: string[] = [];
  private fields = '*';
  private limitVal?: number;

  from(table: string): this {
    this.table = table;
    return this;
  }

  select(...fields: string[]): this {
    this.fields = fields.join(', ');
    return this;
  }

  where(condition: string): this {
    this.conditions.push(condition);
    return this;
  }

  limit(n: number): this {
    this.limitVal = n;
    return this;
  }

  build(): string {
    let query = `SELECT ${this.fields} FROM ${this.table}`;
    if (this.conditions.length > 0) {
      query += ` WHERE ${this.conditions.join(' AND ')}`;
    }
    if (this.limitVal !== undefined) {
      query += ` LIMIT ${this.limitVal}`;
    }
    return query;
  }
}

// 메서드 체이닝으로 직관적인 쿼리 생성
const query = new QueryBuilder()
  .from('users')
  .select('id', 'name', 'email')
  .where('age > 18')
  .where('active = true')
  .limit(10)
  .build();
// SELECT id, name, email FROM users WHERE age > 18 AND active = true LIMIT 10
```

## 5. 구조 패턴

### Adapter — 호환되지 않는 인터페이스 연결

```typescript
// 기존 시스템 (변경 불가)
class LegacyPaymentSystem {
  processPayment(amount: number, cardNumber: string): boolean {
    console.log(`레거시: ${cardNumber}로 ${amount}원 결제`);
    return true;
  }
}

// 새 시스템이 기대하는 인터페이스
interface ModernPaymentGateway {
  pay(payload: { amount: number; token: string }): Promise<{ success: boolean }>;
}

// Adapter: 레거시를 새 인터페이스로 감싸기
class LegacyPaymentAdapter implements ModernPaymentGateway {
  constructor(private legacy: LegacyPaymentSystem) {}

  async pay(payload: { amount: number; token: string }) {
    const success = this.legacy.processPayment(payload.amount, payload.token);
    return { success };
  }
}

// 사용 코드는 레거시를 전혀 모름
const gateway: ModernPaymentGateway = new LegacyPaymentAdapter(
  new LegacyPaymentSystem()
);
await gateway.pay({ amount: 10000, token: 'card_xxx' });
```

### Decorator — 동적으로 기능 추가

```typescript
interface TextFormatter {
  format(text: string): string;
}

class PlainText implements TextFormatter {
  format(text: string) { return text; }
}

// 데코레이터 기반 클래스
abstract class TextDecorator implements TextFormatter {
  constructor(protected wrapped: TextFormatter) {}
  abstract format(text: string): string;
}

class BoldDecorator extends TextDecorator {
  format(text: string) {
    return `<b>${this.wrapped.format(text)}</b>`;
  }
}

class ItalicDecorator extends TextDecorator {
  format(text: string) {
    return `<i>${this.wrapped.format(text)}</i>`;
  }
}

class ColorDecorator extends TextDecorator {
  constructor(wrapped: TextFormatter, private color: string) {
    super(wrapped);
  }
  format(text: string) {
    return `<span style="color:${this.color}">${this.wrapped.format(text)}</span>`;
  }
}

// 자유롭게 조합
const formatter = new ColorDecorator(
  new BoldDecorator(new ItalicDecorator(new PlainText())),
  'blue'
);
console.log(formatter.format('Hello'));
// <span style="color:blue"><b><i>Hello</i></b></span>
```

### Facade — 복잡한 서브시스템을 단순화

```typescript
// 복잡한 서브시스템들
class VideoEncoder { encode(file: string) { /* ... */ } }
```

## 6. 행동 패턴

### Observer — 이벤트 기반 알림

```typescript
interface Observer<T> {
  update(data: T): void;
}

class EventEmitter<T> {
  private observers: Observer<T>[] = [];

  subscribe(observer: Observer<T>): void {
    this.observers.push(observer);
  }

  unsubscribe(observer: Observer<T>): void {
    this.observers = this.observers.filter(o => o !== observer);
  }

  protected emit(data: T): void {
    this.observers.forEach(o => o.update(data));
  }
}

interface StockData {
  symbol: string;
  price: number;
}

class StockMarket extends EventEmitter<StockData> {
  updatePrice(symbol: string, price: number) {
    this.emit({ symbol, price });
  }
}

class PriceAlertSystem implements Observer<StockData> {
  constructor(private threshold: number) {}

  update(data: StockData) {
    if (data.price > this.threshold) {
      console.log(`경고: ${data.symbol} 가격이 ${data.price}원을 초과했습니다!`);
    }
  }
}

class PriceLogger implements Observer<StockData> {
  update(data: StockData) {
    console.log(`[LOG] ${data.symbol}: ${data.price}원`);
  }
}

const market = new StockMarket();
market.subscribe(new PriceAlertSystem(50000));
market.subscribe(new PriceLogger());
market.updatePrice('SAMSUNG', 75000); // 두 옵저버 모두 호출
```

### Strategy — 알고리즘 교체 가능하게

```typescript
interface SortStrategy {
  sort(data: number[]): number[];
}

class BubbleSort implements SortStrategy {
  sort(data: number[]): number[] {
    const arr = [...data];
    for (let i = 0; i < arr.length; i++) {
      for (let j = 0; j < arr.length - i - 1; j++) {
        if (arr[j] > arr[j + 1]) [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
      }
    }
    return arr;
  }
}

class QuickSort implements SortStrategy {
  sort(data: number[]): number[] {
    if (data.length <= 1) return data;
    const pivot = data[Math.floor(data.length / 2)];
    const left = data.filter(x => x < pivot);
    const right = data.filter(x => x > pivot);
    return [...this.sort(left), pivot, ...this.sort(right)];
  }
}

class DataProcessor {
  constructor(private sortStrategy: SortStrategy) {}

  setStrategy(strategy: SortStrategy) {
    this.sortStrategy = strategy;
  }

  process(data: number[]): number[] {
    return this.sortStrategy.sort(data);
  }
}

const processor = new DataProcessor(new BubbleSort());
const data = [64, 34, 25, 12, 22, 11, 90];

console.log(processor.process(data)); // BubbleSort 사용

// 런타임에 전략 교체
processor.setStrategy(new QuickSort());
console.log(processor.process(data)); // QuickSort 사용
```

### Command — 요청을 객체로 캡슐화

```typescript
interface Command {
  execute(): void;
  undo(): void;
}

class TextEditor {
  private content = '';

  insert(text: string, position: number) {
    this.content = this.content.slice(0, position) + text + this.content.slice(position);
  }

  delete(position: number, length: number) {
    this.content = this.content.slice(0, position) + this.content.slice(position + length);
  }

  getContent() { return this.content; }
}

class InsertCommand implements Command {
  constructor(
    private editor: TextEditor,
    private text: string,
    private position: number
  ) {}

  execute() { this.editor.insert(this.text, this.position); }
  undo() { this.editor.delete(this.position, this.text.length); }
}

class CommandHistory {
  private history: Command[] = [];

  execute(command: Command) {
    command.execute();
    this.history.push(command);
  }

  undo() {
    const command = this.history.pop();
    command?.undo();
  }
}

const editor = new TextEditor();
const history = new CommandHistory();

history.execute(new InsertCommand(editor, 'Hello', 0));
history.execute(new InsertCommand(editor, ' World', 5));
console.log(editor.getContent()); // "Hello World"

history.undo();
console.log(editor.getContent()); // "Hello"
```

## 7. 현대적인 아키텍처 패턴

### Clean Architecture

Robert C. Martin이 제안한 아키텍처로, **의존성 규칙**이 핵심입니다.

```
┌─────────────────────────────────────┐
│          Frameworks & Drivers        │  ← 외부: UI, DB, Web
│  ┌───────────────────────────────┐  │
```

```typescript
// Entity (가장 안쪽 레이어 — 순수한 비즈니스 규칙)
class User {
  constructor(
    public readonly id: string,
    public readonly email: string,
    private password: string
  ) {
    if (!email.includes('@')) throw new Error('유효하지 않은 이메일');
  }

  verifyPassword(input: string): boolean {
    return this.password === input; // 실제로는 bcrypt 등
  }
}

// Use Case (Application Business Rules)
interface UserRepository {
  findByEmail(email: string): Promise<User | null>;
}

class LoginUseCase {
  constructor(private userRepo: UserRepository) {}

  async execute(email: string, password: string): Promise<User> {
    const user = await this.userRepo.findByEmail(email);
    if (!user) throw new Error('사용자를 찾을 수 없습니다');
    if (!user.verifyPassword(password)) throw new Error('비밀번호 오류');
    return user;
  }
}

// Interface Adapter (Controller)
class AuthController {
  constructor(private loginUseCase: LoginUseCase) {}

  async login(req: { body: { email: string; password: string } }) {
    try {
      const user = await this.loginUseCase.execute(
        req.body.email,
        req.body.password
      );
      return { status: 200, data: { userId: user.id } };
    } catch (e) {
      return { status: 401, error: (e as Error).message };
    }
  }
}
```

## 8. 실전 적용 가이드

### 언제 어떤 패턴을 쓸까?

| 상황 | 권장 패턴 |
|------|-----------|
| 객체 생성이 복잡할 때 | Builder, Factory Method |
| 전역 상태가 필요할 때 | Singleton (주의해서) |
| 호환되지 않는 인터페이스 | Adapter |
| 기능을 동적으로 추가 | Decorator |
| 이벤트/알림 시스템 | Observer |
| 알고리즘 교체가 필요 | Strategy |
| 실행 취소 기능 | Command |
| 복잡한 서브시스템 단순화 | Facade |
| 데이터 접근 추상화 | Repository |

### 패턴 남용 주의사항

```
❌ 패턴을 위한 패턴
   "이 코드에 Factory를 적용하면 더 멋있어 보일 것 같아"

✅ 문제를 위한 패턴
   "이 코드는 새로운 결제 방식이 생길 때마다 조건문이 늘어나는 문제가 있어
    → Strategy 패턴으로 해결하자"
```

### React에서의 패턴 활용

```typescript
// Compound Component Pattern (구조 패턴의 응용)
const Select = {
  Root: ({ children, value, onChange }: SelectRootProps) => (
    <SelectContext.Provider value={{ value, onChange }}>
      <div className="select-root">{children}</div>
    </SelectContext.Provider>
  ),
  Trigger: ({ children }: { children: ReactNode }) => {
    const { value } = useContext(SelectContext);
    return <button>{children ?? value}</button>;
  },
  Option: ({ value, children }: OptionProps) => {
    const { onChange } = useContext(SelectContext);
    return <li onClick={() => onChange(value)}>{children}</li>;
  },
};

// 사용
<Select.Root value={selected} onChange={setSelected}>
  <Select.Trigger />
  <Select.Option value="ko">한국어</Select.Option>
  <Select.Option value="en">English</Select.Option>
</Select.Root>
```

### 아키텍처 선택 기준

```
프로젝트 규모별 권장 아키텍처

소규모 (1-3인, 단기):
  → 단순한 컴포넌트 분리 + Repository 패턴 정도로 충분

중규모 (팀, 6개월+):
  → MVC/MVVM + SOLID 원칙 적용
  → 레이어드 아키텍처 (Presentation / Business / Data)

대규모 (다수 팀, 장기 운영):
  → Clean Architecture 또는 Hexagonal Architecture
  → 도메인 주도 설계(DDD)와 결합
  → 마이크로서비스 고려
```

---

아키텍처와 디자인 패턴은 **한번에 다 외울 필요가 없습니다**. 실제 문제에 부딪혔을 때 "이런 패턴이 있었지" 하고 떠올리고, 필요할 때 찾아서 적용하는 것이 중요합니다. 중요한 건 **패턴의 이름보다 그 뒤에 있는 원칙** — 변경을 최소화하고, 의존성을 관리하고, 책임을 분리하는 사고방식입니다.

좋은 코드는 처음에 완벽하게 설계된 코드가 아니라, **변화에 유연하게 대응할 수 있는** 코드입니다.
