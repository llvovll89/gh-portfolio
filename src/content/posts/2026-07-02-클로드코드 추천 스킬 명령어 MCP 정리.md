Claude Code는 단순히 "코드를 대신 짜주는 CLI"를 넘어, 스킬(Skill)과 슬래시 명령어(Slash Command), MCP(Model Context Protocol) 서버를 조합해 실제 개발 워크플로우 전체를 자동화할 수 있는 도구로 발전했다. 문제는 기능이 많아질수록 "이걸 언제 어떤 상황에 써야 하는가"를 판단하기가 어려워진다는 점이다. 이 글에서는 실무에서 특히 유용한 것들을 카테고리별로 정리하고, 각각을 언제 꺼내 써야 하는지 기준을 제시한다.

> 목표: 스킬/명령어/MCP의 개념 차이를 이해하고, 상황별로 무엇을 선택해야 할지 판단 기준을 세운다.

## 1. 스킬, 명령어, MCP는 각각 무엇이 다른가

세 가지를 혼동하기 쉬운데, 역할이 명확히 다르다.

| 구분 | 정의 | 예시 |
|------|------|------|
| **슬래시 명령어** | `/foo` 형태로 특정 워크플로우를 즉시 실행하는 단축키 | `/commit`, `/review` |
| **스킬(Skill)** | 특정 작업에 대한 절차와 판단 기준을 담은 지침서. 명령어로도, 자동 트리거로도 호출됨 | `code-review`, `deep-research` |
| **MCP 서버** | 외부 시스템(깃허브, Figma, Notion 등)과 실제로 통신하는 도구 모음 | `github`, `figma` |

즉, **명령어는 진입점**, **스킬은 절차/판단 로직**, **MCP는 외부 세계와의 연결 통로**라고 이해하면 된다. 예를 들어 `/review`라는 명령어를 치면 내부적으로 `review` 스킬이 실행되고, 그 스킬이 GitHub PR 정보를 가져올 때 `mcp__github__*` 도구를 호출하는 식이다.

## 2. 코드 품질 관련 추천 명령어

### `/code-review`

현재 diff를 대상으로 정확성 버그와 재사용/단순화/효율성 이슈를 찾아준다. effort 레벨을 조절할 수 있다.

```bash
/code-review              # 기본 레벨
/code-review --fix        # 발견한 이슈를 워킹 트리에 바로 적용
/code-review --comment    # PR에 인라인 코멘트로 등록
```

- **low/medium**: 확신도 높은 소수의 이슈만 리포트 (빠른 셀프 체크에 적합)
- **high~max**: 넓은 커버리지, 불확실한 이슈까지 포함 (배포 전 꼼꼼한 검토에 적합)

### `/simplify`

버그를 찾는 게 아니라 **품질(재사용성, 단순화, 효율성)만** 검토하고 바로 수정까지 적용한다. `/code-review`가 "찾아서 보고"라면 `/simplify`는 "찾아서 고침"에 가깝다.

### `/security-review`

현재 브랜치의 변경사항에 대해 보안 취약점(OWASP Top 10 등)을 점검한다. PR을 올리기 전 습관적으로 돌려볼 가치가 있다.

### `verify` 스킬

코드 변경이 "실제로 의도대로 동작하는지"를 엔드투엔드로 검증한다. 테스트나 타입체크로는 못 잡는, 실제 실행 흐름의 문제를 잡아낸다. 단, 테스트/문서만 바뀐 diff에는 적용할 실익이 없다.

## 3. Git / PR 워크플로우 명령어

### `commit` 스킬

현재 변경사항을 분석해서 상세한 커밋 메시지를 작성하고 커밋한다. 커밋 메시지 컨벤션을 매번 고민하지 않아도 된다.

```bash
/commit
```

### `review` 스킬 (GitHub PR 리뷰)

로컬 diff가 아니라 **이미 올라간 GitHub PR**을 리뷰할 때 쓴다. `mcp__github__get_pull_request_files` 등을 통해 PR 내용을 가져와 분석한다.

```bash
/review https://github.com/org/repo/pull/123
```

> 팁: 로컬 작업 중인 diff는 `/code-review`, 이미 push된 PR은 `review` 스킬로 구분해서 쓰면 헷갈리지 않는다.

## 4. 멀티 에이전트 오케스트레이션

### Agent 도구

단일 작업을 서브에이전트에게 위임할 때 사용한다. `Explore`(읽기 전용 탐색), `general-purpose`(범용), `Plan`(설계) 등 특화된 에이전트 타입을 상황에 맞게 고른다.

```
"이 리포지토리에서 인증 로직이 어디 흩어져 있는지 찾아줘"
→ Explore 에이전트에 위임 (검색 전용이라 빠르고 안전)
```

### Workflow 도구

여러 에이전트를 **결정적으로(deterministic)** 파이프라인/병렬로 조율해야 할 때 쓴다. 단, 명시적으로 "워크플로우로 처리해줘", "멀티 에이전트로 돌려줘" 같은 opt-in이 있을 때만 사용하는 게 원칙이다. 토큰 소모가 크기 때문에 사용자 동의 없이 임의로 돌리지 않는다.

대표 패턴:
- **Adversarial verify**: 여러 에이전트가 서로의 발견을 반박하게 해서 오탐을 걸러냄
- **Loop-until-dry**: 더 이상 새로운 이슈가 안 나올 때까지 반복 탐색
- **Judge panel**: 여러 접근법을 독립적으로 만들고 점수를 매겨 종합

## 5. 디자인 연동 MCP: Figma

프론트엔드 작업에서 디자인-코드 왕복이 잦다면 Figma MCP 서버가 특히 유용하다.

| 방향 | 용도 | 관련 스킬 |
|------|------|-----------|
| Figma → 코드 | 디자인을 읽어서 실제 컴포넌트로 구현 | `figma-generate-design` |
| 코드 → Figma | 기존 코드/페이지를 Figma에 역으로 반영 | `figma-generate-design`, `create_new_file` |
| 컴포넌트 매핑 | Figma 컴포넌트와 코드 컴포넌트를 연결 | `figma-code-connect` |
| 디자인 시스템 구축 | 변수/토큰/컴포넌트 라이브러리 생성 | `figma-generate-library` |

핵심 규칙: **`use_figma`나 `create_new_file`을 직접 호출하기 전에 반드시 해당 전용 스킬(`figma-use`, `figma-create-new-file`)을 먼저 로드**해야 한다. 스킬을 건너뛰면 디버깅하기 어려운 실패가 자주 발생한다.

## 6. 리서치 및 문서화 스킬

### `deep-research`

여러 웹 검색을 팬아웃으로 실행하고, 출처를 교차 검증한 뒤 인용이 달린 리포트를 만든다. 질문이 너무 광범위하면(`"어떤 차를 사야 할까?"`) 먼저 예산/용도 등을 되물어서 범위를 좁힌 뒤 진행한다.

```
/deep-research "2026년 상반기 프론트엔드 번들러 트렌드 비교"
```

### `post` 스킬 (이 글도 이걸로 작성됨)

`src/content/posts/`에 마크다운 블로그 포스트를 생성한다. 제목만 던지면 프론트매터, 목차, 섹션까지 구조화해서 작성해준다.

### `init` / `claude-md-improver`

`CLAUDE.md`를 새로 만들거나(`init`), 기존 문서의 품질을 감사하고 개선한다(`claude-md-improver`). 프로젝트 컨벤션이 바뀌었는데 `CLAUDE.md`가 방치되어 있다면 후자를 정기적으로 돌리는 게 좋다.

## 7. 프로젝트 설정 및 상태 관리

### `update-config` 스킬

"앞으로 X할 때마다 Y해줘" 같은 자동화 요청은 메모리로 해결되지 않는다. 이런 요청은 반드시 `settings.json`의 훅(hook)으로 등록해야 하며, 이때 `update-config` 스킬을 사용한다.

```
"커밋할 때마다 lint 자동 실행해줘" → update-config 스킬이 pre-commit 훅 등록
```

### `fewer-permission-prompts`

반복적으로 승인해야 하는 읽기 전용 Bash/MCP 명령어를 트랜스크립트에서 스캔해서, `.claude/settings.json`에 허용 목록을 자동으로 추가해준다. 매번 같은 명령어에 승인 버튼을 누르는 게 피곤하다면 한 번 돌려볼 만하다.

### `/loop`, `schedule`

- `/loop 5m /deploy-check`: 특정 명령어를 주기적으로 반복 실행
- `schedule`: cron 기반으로 클라우드에서 정기 실행되는 에이전트(루틴)를 등록

## 8. 나만의 스킬 만들기

팀/프로젝트 반복 작업이 있다면 커스텀 스킬로 만들어두는 게 장기적으로 이득이다. `.claude/agents/*.md` 형태로 정의하며, frontmatter에 이름/설명/트리거 조건을 명시한다. 이 글의 `post` 스킬처럼 "블로그 포스트 작성"이라는 반복 작업을 스킬로 만들어두면, 다음부터는 제목만 말해도 동일한 품질의 결과물이 나온다.

## 9. 상황별 추천 조합 정리

| 상황 | 추천 조합 |
|------|-----------|
| PR 올리기 전 마지막 점검 | `/code-review` → `/security-review` → `verify` |
| 이미 올라간 PR 리뷰 | `review` 스킬 + `mcp__github__*` |
| 디자인 시안을 코드로 구현 | `figma-use` + `figma-generate-design` |
| 반복되는 배포/체크 자동화 | `update-config`로 훅 등록 또는 `schedule` |
| 애매한 주제 리서치 | `deep-research` (범위 좁히는 질문부터) |
| 대규모 코드 감사/마이그레이션 | `Workflow` 도구 (사용자 명시 승인 시에만) |
| 매번 승인 팝업이 귀찮을 때 | `fewer-permission-prompts` |

결론적으로, Claude Code를 잘 쓴다는 것은 기능을 많이 아는 것이 아니라 **"지금 이 상황에 어떤 도구의 책임 범위가 맞는가"**를 빠르게 판단하는 것이다. 명령어는 빠른 실행, 스킬은 절차의 일관성, MCP는 외부 연동을 담당한다는 큰 그림만 기억해도 대부분의 선택은 쉬워진다.
