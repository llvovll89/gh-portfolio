백엔드는 사용자에게 직접 보이지 않지만, 서비스의 신뢰성과 성능을 결정하는 핵심 영역입니다. "요청이 들어와서 응답이 나가는" 단순한 한 줄 뒤에는 라우팅, 인증, 비즈니스 로직, 트랜잭션, 캐싱, 비동기 처리까지 수많은 레이어가 있습니다. 이 글에서는 백엔드 전체 프로세스 흐름을 기반으로 각 구간의 필수 개념을 체계적으로 정리합니다.

> 목표: HTTP 요청이 들어오는 순간부터 응답이 반환되기까지의 전체 백엔드 프로세스를 이해하고, 각 단계에서 반드시 알아야 할 개념과 실무 패턴을 습득한다.

## 1. 백엔드 전체 처리 흐름 개요

클라이언트가 `https://api.example.com/users/1`을 요청했을 때 벌어지는 일을 순서대로 나열하면 다음과 같습니다.

```
[클라이언트]
    │
    ▼ DNS 조회 → IP 획득
    │
    ▼ TCP 3-way Handshake
    │
    ▼ TLS Handshake (HTTPS)
    │
    ▼ HTTP 요청 전송
    │
[웹 서버 / 리버스 프록시 (Nginx)]
    │  - SSL 종료
    │  - 정적 파일 서빙
    │  - 로드 밸런싱
    ▼
[WAS / 애플리케이션 서버 (Spring Boot / Node.js)]
    │
    ▼ 미들웨어 파이프라인
    │  - 로깅
    │  - 인증(JWT 검증)
    │  - Rate Limiting
    │
    ▼ 라우터 → 컨트롤러
    │
    ▼ 서비스 레이어 (비즈니스 로직)
    │
    ▼ 리포지토리 레이어
    │
    ▼ DB 쿼리 (캐시 히트 시 Redis에서 반환)
    │
    ▼ 응답 직렬화 (JSON)
    │
[클라이언트] ← HTTP 응답 반환
```

각 단계가 독립적인 책임을 가지므로, 장애나 병목이 발생했을 때 어느 레이어인지 파악하는 것이 디버깅의 핵심입니다.

## 2. 네트워크 레이어: DNS, TCP, TLS

### DNS (Domain Name System)

도메인을 IP 주소로 변환하는 시스템입니다. 브라우저는 OS 캐시 → 로컬 DNS 서버 → 루트 → TLD → 권한 DNS 순으로 조회합니다.

| 레코드 타입 | 용도 |
|---|---|
| A | 도메인 → IPv4 |
| AAAA | 도메인 → IPv6 |
| CNAME | 도메인 → 다른 도메인 (별칭) |
| MX | 메일 서버 |
| TXT | SPF, DKIM 등 검증용 텍스트 |

### TCP 3-way Handshake

```
Client → SYN → Server
Client ← SYN-ACK ← Server
Client → ACK → Server
[연결 완료 — 이후 데이터 전송]
```

TCP는 신뢰성 있는 전송을 보장하지만, 매번 핸드셰이크 비용이 발생합니다. HTTP/2는 멀티플렉싱으로, HTTP/3(QUIC)는 UDP 기반으로 이 비용을 줄였습니다.

### TLS (Transport Layer Security)

HTTPS의 기반. 인증서 검증 → 키 교환 → 대칭키로 암호화 통신을 수행합니다.

- **TLS 1.2**: 2-RTT (왕복 2번)
- **TLS 1.3**: 1-RTT, 0-RTT 재연결 지원으로 훨씬 빠름

## 3. 웹 서버와 WAS 역할 분리

| 구분 | 웹 서버 (Nginx, Apache) | WAS (Spring Boot, Node.js) |
|---|---|---|
| 역할 | 정적 파일 서빙, SSL 종료, 프록시 | 동적 비즈니스 로직 처리 |
| 처리 방식 | 이벤트 기반 (비동기) | 스레드 풀 or 이벤트 루프 |
| 확장성 | 수평 확장 용이 | 애플리케이션 복잡도에 따라 다름 |

### Nginx 리버스 프록시 예시

```nginx
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /static/ {
        root /var/www;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

**X-Forwarded-For** 헤더: 리버스 프록시 뒤에서 실제 클라이언트 IP를 추적할 때 사용합니다.

## 4. 라우팅과 미들웨어 파이프라인

요청은 미들웨어 파이프라인을 순서대로 통과한 뒤 컨트롤러에 도달합니다.

### Spring Boot 필터 vs 인터셉터

```
HTTP 요청
    │
    ▼ Filter (javax.servlet.Filter)
    │  - 인증 토큰 파싱
    │  - CORS 처리
    │  - 요청/응답 로깅
    │
    ▼ DispatcherServlet
    │
    ▼ HandlerInterceptor (preHandle)
    │  - 권한 검사
    │  - Rate Limiting
    │
    ▼ Controller
    │
    ▼ HandlerInterceptor (postHandle / afterCompletion)
```

### Node.js (Express) 미들웨어

```javascript
// 순서가 중요 — 위에서 아래로 실행
app.use(morgan('combined'));        // 로깅
app.use(helmet());                  // 보안 헤더
app.use(express.json());            // 바디 파싱
app.use(rateLimiter);               // Rate Limiting
app.use('/api', authMiddleware);    // 인증 (특정 경로만)
app.use('/api/users', userRouter);  // 라우터
app.use(errorHandler);              // 전역 에러 핸들러
```

## 5. 인증과 인가 (Authentication & Authorization)

- **인증(Authentication)**: "너 누구야?" — 신원 확인
- **인가(Authorization)**: "너 이거 해도 돼?" — 권한 확인

### Session vs JWT

| 구분 | Session | JWT |
|---|---|---|
| 상태 저장 | 서버 (세션 스토어) | 클라이언트 (토큰 자체) |
| 확장성 | 서버 간 세션 공유 필요 | Stateless, 수평 확장 유리 |
| 무효화 | 즉시 가능 | 만료 전 무효화 어려움 |
| 보안 | CSRF 취약 | XSS 취약 (localStorage 저장 시) |

### JWT 구조

```
Header.Payload.Signature

eyJhbGciOiJIUzI1NiJ9          ← Header (알고리즘)
.eyJzdWIiOiIxMjMiLCJleHAiOjE3fQ  ← Payload (클레임)
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  ← Signature
```

**Payload 클레임 예시:**

```json
{
  "sub": "user-123",
  "email": "user@example.com",
  "roles": ["ROLE_USER"],
  "iat": 1716000000,
  "exp": 1716086400
}
```

### Access Token + Refresh Token 전략

```
[최초 로그인]
Client → POST /auth/login → Server
Server → Access Token (15분) + Refresh Token (7일)

[API 호출]
Client → Authorization: Bearer {access_token}

[Access Token 만료 시]
Client → POST /auth/refresh { refreshToken }
Server → 새 Access Token 발급

[Refresh Token 탈취 대비]
- Refresh Token Rotation: 사용 시마다 새 RT 발급
- RTR + DB 저장으로 이전 RT 무효화
```

### OAuth 2.0 흐름

```
User → Client → Authorization Server (구글, 카카오)
Authorization Server → Authorization Code → Client
Client → Code + Client Secret → Access Token
Client → Access Token → Resource Server (사용자 정보)
```

## 6. 계층형 아키텍처 (Layered Architecture)

Spring Boot 기준의 전형적인 계층 구조:

```
Presentation Layer (Controller)
    │  HTTP 요청/응답, DTO 변환, 입력 검증
    ▼
Business Layer (Service)
    │  비즈니스 로직, 트랜잭션 경계
    ▼
Persistence Layer (Repository)
    │  DB 접근 (JPA, MyBatis, JDBC)
    ▼
Database
```

### 각 계층의 책임 분리

```java
// Controller — HTTP 레이어만 담당
@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserService userService;

    @GetMapping("/{id}")
    public ResponseEntity<UserResponse> getUser(@PathVariable Long id) {
        return ResponseEntity.ok(userService.getUser(id));
    }
}

// Service — 비즈니스 로직
@Service
@Transactional(readOnly = true)
public class UserService {
    private final UserRepository userRepository;

    public UserResponse getUser(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        return UserResponse.from(user);
    }
}

// Repository — 데이터 접근
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email);
}
```

**DTO(Data Transfer Object)** 를 반드시 사용해야 하는 이유:
- Entity를 직접 반환하면 순환 참조, 불필요한 데이터 노출, 레이어 결합 문제 발생
- DTO로 변환하여 API 스펙과 내부 구현을 분리

## 7. 트랜잭션과 동시성 제어

### ACID 속성

| 속성 | 설명 |
|---|---|
| **Atomicity(원자성)** | 트랜잭션 내 작업은 전부 성공하거나 전부 실패 |
| **Consistency(일관성)** | 트랜잭션 전후 데이터 무결성 유지 |
| **Isolation(격리성)** | 동시 실행 트랜잭션이 서로 간섭 X |
| **Durability(지속성)** | 커밋된 데이터는 시스템 장애 후에도 보존 |

### 격리 수준 (Isolation Level)

```
READ UNCOMMITTED  ← Dirty Read 발생 가능 (거의 미사용)
READ COMMITTED    ← 대부분 DB의 기본값 (PostgreSQL)
REPEATABLE READ   ← MySQL InnoDB 기본값
SERIALIZABLE      ← 가장 엄격, 성능 저하 큼
```

### 동시성 문제와 해결

**낙관적 잠금(Optimistic Lock)** — 충돌이 드물 때

```java
@Entity
public class Product {
    @Version  // JPA가 자동으로 버전 체크
    private Long version;

    private int stock;
}
// 동시 수정 시 ObjectOptimisticLockingFailureException 발생 → 재시도 로직 필요
```

**비관적 잠금(Pessimistic Lock)** — 충돌이 잦을 때

```java
@Query("SELECT p FROM Product p WHERE p.id = :id")
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<Product> findByIdWithLock(Long id);
// SELECT ... FOR UPDATE → 다른 트랜잭션 대기
```

**데드락 방지 원칙**: 여러 리소스를 항상 같은 순서로 잠금 획득.

## 8. 캐싱 전략

캐시는 DB 부하를 줄이고 응답 속도를 높이는 핵심 수단입니다.

### 캐시 위치별 분류

```
[클라이언트]
    - 브라우저 캐시 (Cache-Control, ETag)
[CDN]
    - 정적 자산, 공개 API 응답
[애플리케이션]
    - In-Memory (Caffeine, Guava)
    - 외부 캐시 (Redis, Memcached)
[DB]
    - 쿼리 캐시 (일반적으로 미권장)
```

### 캐시 패턴

**Cache-Aside (Lazy Loading)** — 가장 일반적

```
1. 캐시 조회
2. Hit → 즉시 반환
3. Miss → DB 조회 → 캐시 저장 → 반환
```

```java
public User getUser(Long id) {
    String key = "user:" + id;
    User cached = redisTemplate.opsForValue().get(key);
    if (cached != null) return cached;

    User user = userRepository.findById(id).orElseThrow(...);
    redisTemplate.opsForValue().set(key, user, Duration.ofMinutes(10));
    return user;
}
```

**Write-Through** — 쓰기 시 캐시와 DB 동시 업데이트 (일관성 우선)

**Write-Behind (Write-Back)** — 캐시에 먼저 쓰고 비동기로 DB 반영 (성능 우선, 유실 위험)

### 캐시 무효화 전략

```
TTL(Time-To-Live) 기반: 일정 시간 후 자동 만료
이벤트 기반: 데이터 변경 시 명시적 삭제/갱신
```

**캐시 스탬피드(Thundering Herd)**: TTL 만료 시 다수의 요청이 동시에 DB를 치는 문제. 해결책: Mutex Lock, Probabilistic Early Expiration.

## 9. 비동기 처리와 메시지 큐

동기 처리로는 감당하기 어려운 작업(이메일 발송, 이미지 처리, 대용량 배치)은 메시지 큐로 분리합니다.

### 메시지 큐의 이점

- **디커플링**: Producer와 Consumer가 직접 연결되지 않음
- **버퍼링**: 트래픽 스파이크 흡수
- **재시도**: 실패 시 자동 재처리
- **순서 보장**: FIFO 큐로 처리 순서 보장

### 주요 메시지 큐 비교

| 도구 | 특징 | 적합한 상황 |
|---|---|---|
| **RabbitMQ** | AMQP, 라우팅 유연 | 작업 큐, 복잡한 라우팅 |
| **Apache Kafka** | 높은 처리량, 로그 보존 | 이벤트 스트리밍, 대용량 |
| **Redis Pub/Sub** | 간단, 인메모리 | 실시간 알림, 채팅 |
| **SQS (AWS)** | 관리형, 높은 안정성 | 클라우드 환경 |

### Kafka 기본 개념

```
Producer → Topic(Partition) → Consumer Group
              │
         offset 기반 읽기 (재처리 가능)
         메시지 영속 보존 (기본 7일)
```

```java
// Spring Kafka Producer
kafkaTemplate.send("order-topic", orderId, orderEvent);

// Consumer
@KafkaListener(topics = "order-topic", groupId = "inventory-service")
public void consume(OrderEvent event) {
    inventoryService.decreaseStock(event.getProductId(), event.getQuantity());
}
```

## 10. REST API 설계 원칙

### RESTful URL 규칙

```
✅ 올바른 설계
GET    /api/users          → 목록 조회
GET    /api/users/{id}     → 단건 조회
POST   /api/users          → 생성
PUT    /api/users/{id}     → 전체 수정
PATCH  /api/users/{id}     → 부분 수정
DELETE /api/users/{id}     → 삭제

❌ 잘못된 설계
GET /api/getUsers
POST /api/deleteUser
GET /api/users/create
```

### HTTP 상태 코드 정확히 쓰기

| 코드 | 의미 | 사용 상황 |
|---|---|---|
| 200 OK | 성공 | 조회, 수정 성공 |
| 201 Created | 생성 성공 | POST 후 리소스 생성 |
| 204 No Content | 성공, 응답 바디 없음 | DELETE 성공 |
| 400 Bad Request | 잘못된 요청 | 입력 유효성 실패 |
| 401 Unauthorized | 인증 실패 | 토큰 없음/만료 |
| 403 Forbidden | 권한 없음 | 인증됐지만 접근 불가 |
| 404 Not Found | 리소스 없음 | ID에 해당하는 데이터 없음 |
| 409 Conflict | 충돌 | 이메일 중복 등 |
| 500 Internal Server Error | 서버 오류 | 예상치 못한 예외 |

### API 버전 관리

```
URI 버전:     /api/v1/users, /api/v2/users
헤더 버전:    Accept: application/vnd.api+json;version=1
쿼리 파라미터: /api/users?version=1
```

URI 버전 방식이 가장 직관적이고 널리 사용됩니다.

## 11. 에러 처리와 예외 전략

### 예외 계층 설계

```java
// 커스텀 예외 계층
BusinessException (RuntimeException)
    ├── UserNotFoundException
    ├── DuplicateEmailException
    └── InsufficientStockException
```

### Spring 전역 예외 핸들러

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleUserNotFound(UserNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse("USER_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return ResponseEntity.badRequest()
            .body(new ErrorResponse("VALIDATION_FAILED", message));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception e) {
        log.error("Unhandled exception", e);  // 내부 상세는 로그에만
        return ResponseEntity.internalServerError()
            .body(new ErrorResponse("INTERNAL_ERROR", "서버 오류가 발생했습니다."));
    }
}
```

**핵심 원칙**: 클라이언트에게 내부 스택 트레이스나 DB 오류 메시지를 절대 노출하지 않습니다. 내부 상세는 로그에, 외부에는 코드화된 메시지만.

## 12. 로깅과 모니터링

### 로그 레벨 활용

```
TRACE  — 매우 상세한 디버그 (성능 측정, 메서드 진입/종료)
DEBUG  — 디버깅용 변수 값, 쿼리 파라미터
INFO   — 정상 비즈니스 이벤트 (주문 생성, 로그인 성공)
WARN   — 잠재적 문제 (재시도 발생, 설정값 기본값 사용)
ERROR  — 처리된 오류 (예외 발생, 외부 서비스 실패)
FATAL  — 서비스 불가 상태 (DB 연결 불가)
```

### 구조화 로깅 (Structured Logging)

```java
// 나쁜 예 — 파싱 불가
log.info("User " + userId + " ordered " + productId);

// 좋은 예 — JSON 구조로 검색/집계 가능
log.info("Order created", kv("userId", userId), kv("productId", productId),
         kv("amount", amount), kv("traceId", MDC.get("traceId")));
```

### Distributed Tracing

MSA 환경에서 요청이 여러 서비스를 거칠 때, `Trace ID`를 전파하여 전체 흐름을 추적합니다.

```
Client → Service A → Service B → Service C
         [TraceID: abc123, SpanID: 1]
                      [TraceID: abc123, SpanID: 2]
                                   [TraceID: abc123, SpanID: 3]
```

**도구 스택 예시:**
- **Zipkin / Jaeger**: 분산 트레이싱 수집
- **ELK Stack** (Elasticsearch + Logstash + Kibana): 로그 수집 및 시각화
- **Prometheus + Grafana**: 메트릭 수집 및 대시보드
- **Sentry**: 에러 트래킹

## 정리

| 구간 | 핵심 개념 |
|---|---|
| 네트워크 | DNS, TCP 핸드셰이크, TLS, HTTP/2 |
| 서버 진입 | Nginx 리버스 프록시, 로드 밸런싱 |
| 미들웨어 | 필터, 인터셉터, 인증/인가 파이프라인 |
| 인증 | JWT, Session, OAuth 2.0, Refresh Token Rotation |
| 아키텍처 | 계층형 구조, DTO 분리, 단일 책임 원칙 |
| DB | ACID, 격리 수준, 낙관적/비관적 잠금 |
| 캐싱 | Cache-Aside, TTL, 캐시 스탬피드 |
| 비동기 | Kafka, RabbitMQ, 이벤트 기반 아키텍처 |
| API 설계 | RESTful URI, 올바른 HTTP 상태 코드, 버전 관리 |
| 에러 처리 | 커스텀 예외 계층, 전역 핸들러, 외부 노출 금지 |
| 관찰 가능성 | 구조화 로깅, Distributed Tracing, 메트릭 |

백엔드는 각 레이어가 명확한 책임을 가지고, 서로 느슨하게 결합될수록 유지보수와 확장이 쉬워집니다. 요청 흐름 전체를 머릿속에 그릴 수 있는 것 — 그것이 백엔드 개발자의 기본기입니다.
