2025년은 AI 에이전트가 코드를 '쓸 수 있음'을 증명한 해였다. 그리고 2026년, 업계는 한 가지 사실을 깨달았다.

> **"에이전트 자체가 어려운 게 아니라, 하네스(Harness)가 어렵다."** — Aakash Gupta

Stripe는 AI 에이전트로 주당 1,300건 이상의 PR을 자동 머지한다. OpenAI는 100만 줄이 넘는 코드베이스를 에이전트로 유지보수한다. 이 모든 것이 가능한 이유는 모델의 성능이 아니라, 모델을 감싸는 **하네스(Harness)** 덕분이다.

## 1. 하네스(Harness)란 무엇인가?

### 세 가지 정의

| 출처 | 정의 |
|------|------|
| **Salesforce** | AI의 도구, 메모리, 안전성을 관리하여 신뢰할 수 있는 자율 작업 실행을 보장하는 **운영 소프트웨어 계층** |
| **Martin Fowler (Birgitta Böckeler)** | AI 에이전트가 코드를 대규모로 생성·유지하도록 돕는 **제약, 피드백 루프, 문서화, 린터, 수명주기 관리 시스템**의 총체 |
| **Phil Schmid (Hugging Face)** | 컨텍스트 관리, 부팅 시퀀스, 표준 드라이버를 담당하는 — AI 에이전트의 **운영체제(OS)** |

쉽게 말하면, 하네스는 **"모델을 감싸는 인프라 구조"**다. 모델이 아무리 강력해도, 하네스 없이는 프로덕션에서 안전하게 동작하지 않는다.

### 어원

"Harness"는 말(馬)의 마구(馬具) — 고삐·안장·재갈의 집합체 — 에서 유래했다. 강력하지만 예측 불가능한 동물을 올바른 방향으로 이끌기 위한 장비처럼, AI 에이전트도 강력하지만 제어가 필요하다.

### 하네스 = AI 에이전트의 OS

| 컴퓨터 구성 요소 | AI 에이전트 대응 |
|------|------|
| CPU | 모델 (추론 능력) |
| RAM | 컨텍스트 윈도우 (작업 메모리) |
| **운영체제** | **Harness** |
| 애플리케이션 | Agent (사용자 목적 로직) |

### 왜 필요한가?

하네스 없이 에이전트를 실행하면 다음 문제가 발생한다.

| 문제 | 증상 |
|------|------|
| **Context Drift** | 에이전트가 원래 목표를 잊어버리는 "AI 건망증" |
| **무한 루프** | 자기 수정 조건 없이 같은 실패 반복 |
| **도구 오류** | 통제 없는 외부 API 호출로 부작용 발생 |
| **자원 폭발** | 비용·시간이 예측 불가능하게 증가 |
| **보안 취약** | 프롬프트 인젝션, 코드 실행 취약점 노출 |

## 2. 하네스의 핵심 구성 요소 7가지

### ① Context Engineering (컨텍스트 공학)

에이전트가 "컨텍스트 안에서 볼 수 없는 것은 존재하지 않는다"는 원칙이 핵심이다.

- **동적 컨텍스트 주입:** 관찰 데이터, 브라우저 탐색, 외부 지식 베이스를 실시간 삽입
- **컨텍스트 컴팩션(Context Compaction):** 컨텍스트 윈도우 한계 도달 전 자동 요약·압축
- **세션 간 상태 보존:** `claude-progress.txt`, Git 커밋 메시지, JSON 기능 목록 등으로 다음 세션 재구성

### ② Tool Orchestration (도구 오케스트레이션)

- 도구 요청을 **가로채고 → 검증하고 → 실행하고 → 결과를 정화(sanitize)**하는 미들웨어 역할
- **MCP(Model Context Protocol):** Anthropic이 2024년 11월 발표한 에이전트-도구 통합 오픈 표준
  - 2025년 3월 OpenAI 공식 채택
  - Stripe: MCP 서버 한 곳에 **400개 이상** 내부 도구 집중 관리

### ③ Memory System (메모리 시스템)

4가지 메모리 유형을 목적에 맞게 조합한다.

| 유형 | 설명 | 예시 |
|------|------|------|
| **In-context** | 현재 컨텍스트 윈도우 내 일시 기억 | 현재 태스크 진행 상황 |
| **External** | 영구 저장소 | 벡터 DB, 파일 시스템 |
| **In-weights** | 파인튜닝으로 학습된 지식 | 도메인 특화 모델 |
| **In-cache** | KV 캐시를 통한 중간 상태 보존 | 반복 요청 최적화 |

### ④ Planning & Decomposition (계획 및 분해)

복잡한 목표를 단계적 하위 작업으로 분해하고, 태스크 복잡도에 따라 서브에이전트 수와 도구 호출 횟수를 자동 조정한다.

| 태스크 복잡도 | 서브에이전트 수 | 도구 호출 횟수 |
|------|------|------|
| 단순 사실 조회 | 1개 | 3~10회 |
| 비교 분석 | 2~4개 | 각 10~15회 |
| 복잡한 연구·구현 | 10개 이상 | 수백 회 |

### ⑤ Safety & Guardrails (안전 및 가드레일)

- **최소 권한 원칙(Principle of Least Privilege):** 현재 태스크에 필요한 것만 정확히 접근
- **Human-in-the-Loop(HITL):** 금융 거래, 데이터 삭제, 접근 제어 변경 시 인간 승인 강제
- **결정론적 게이트:** 린터, CI 검증, 구조 테스트 등 에이전트가 건너뛸 수 없는 자동화 단계
- **최대 반복 횟수 제한:** 무한 루프 방지

### ⑥ State Management & Persistence (상태 관리)

- 에이전트 진행 상황을 DB에 저장 → 실패 시 재시작 없이 복구 가능
- Git 히스토리 + 진행 문서로 다음 세션 컨텍스트 재구성
- 기능 추적 JSON으로 완료/미완료 항목 명시

### ⑦ Evaluation & Verification (평가 및 검증)

| 방식 | 내용 |
|------|------|
| **Rule-based feedback** | 코드 린팅, 유닛 테스트, 유효성 검사 |
| **Visual feedback** | 스크린샷, 렌더링 결과 확인 |
| **LLM-as-judge** | 주관적 평가 기준에 대한 별도 LLM 판정 |

## 4. Anthropic의 하네스 설계 전략

### 핵심 설계 철학

> **"Claude는 프로그래머가 매일 사용하는 것과 동일한 도구가 필요하다."**

파일 탐색, 파일 편집, 린팅, 실행, 디버깅을 반복적으로 수행하는 4단계 피드백 루프가 기본이다.

```
컨텍스트 수집 → 액션 수행 → 결과 검증 → 반복
```

### 장기 실행 에이전트 하네스 구조

Anthropic이 공개한 이중 에이전트 시스템:

```
초기화 에이전트 (Initializer Agent)
  └─ 환경 세팅 (init.sh 실행, 상태 파일 초기화)

코딩 에이전트 (Coding Agent)
  └─ 반복 세션 실행
  └─ claude-progress.txt 기록
  └─ Git 커밋으로 진행 상황 보존
  └─ features.json으로 pass/fail 추적
```

### 멀티에이전트 연구 시스템 (2025)

- **리드 에이전트(Claude Opus 4)** + **서브에이전트(Claude Sonnet 4)** 구조
- 리드 에이전트가 쿼리 분석 → 전략 수립 → 서브에이전트 병렬 생성
- 결과: 단일 에이전트 Claude Opus 4 대비 내부 연구 평가에서 **90.2% 성능 향상**

### "Building Effective Agents" 5가지 워크플로우 (2024년 12월 발표)

Anthropic이 공식화한 에이전트 설계 패턴:

1. **Prompt Chaining** — 순차적 단계 처리
2. **Routing** — 입력 분류 후 전문 핸들러로 전달
3. **Parallelization** — 독립 태스크 병렬 실행
4. **Orchestrator-Subagents** — 동적 태스크 분해 및 위임
5. **Evaluator-Optimizer** — 생성 + 반복 평가·개선

## 5. 안전성과 하네스 — 가드레일 설계

### 샌드박싱 아키텍처 (2026 기준 권장)

| 방식 | 격리 수준 | 적합 사례 |
|------|------|------|
| **MicroVMs** (Firecracker, Kata) | 가장 강력, 전용 커널 | 신뢰할 수 없는 코드 실행 |
| **gVisor** (사용자 공간 커널) | 중간, 시스템 콜 가로채기 | 컴퓨팅 집약적 워크로드 |
| **표준 컨테이너** | 가장 약함, 공유 커널 취약 | 검증된 신뢰 코드에만 |

**권장 조합:**
- 에이전트 실행 → `gVisor / GKE Sandbox`
- 코드 실행 도구 → `Firecracker MicroVM`
- 플러그인 → `WASI 기능 범위 제한`
- 네트워크 → **이그레스 기본 차단**, 필요 엔드포인트만 허용 목록

### 권한 제어 원칙

```
최소 권한 (Least Privilege)
  ├─ 현재 태스크에 필요한 접근만 허용
  ├─ 단기 자격증명 (태스크별 임시 발급)
  └─ ABAC/RBAC (OPA 기반 정책 평가)

고위험 액션 이중 승인 (HITL)
  ├─ 재무 영향 작업
  ├─ 데이터 삭제·복구 불가 작업
  └─ 접근 제어 정책 변경
```

### OWASP AI 에이전트 보안 Top 5 (2026)

1. **프롬프트 인젝션 (Prompt Injection)**
2. **컨텍스트 오염 (Context Poisoning)**
3. **도구 남용 (Tool Abuse)**
4. **코드 생성 익스플로잇**
5. **메모리 오염 공격**

## 6. 하네스 설계 패턴 6가지

### ① Orchestrator-Subagent 패턴

중앙 LLM이 태스크를 동적으로 분해하여 워커 LLM에게 위임하고 결과를 합성한다. 서브태스크가 사전 정의되지 않고 맥락에 따라 결정된다는 점에서 단순 병렬화와 구별된다.

```
User Query
    │
    ▼
Lead Agent (Claude Opus 4)
  쿼리 분석 → 전략 수립
    │         │         │
    ▼         ▼         ▼
SubAgent1  SubAgent2  SubAgent3
(Sonnet 4) (Sonnet 4) (Sonnet 4)
    │         │         │
    └────┬────┘
         ▼
Lead Agent (합성·요약)
         │
         ▼
  Final Response
```

### ② Deterministic Gate 패턴

창의적 LLM 단계와 결정론적 단계를 교차 배치하여 에이전트가 검증 단계를 건너뛸 수 없도록 강제한다.

```
에이전트 → 코드 작성 (창의적)
         → 린터 강제 실행 ← [GATE: 에이전트 건너뛰기 불가]
         → 린팅 결과 수정 (창의적)
         → 테스트 강제 실행 ← [GATE]
         → PR 오픈
```

### ③ Human-in-the-Loop 패턴

고위험 액션에서 실행을 일시 중단하고 인간 승인을 요구한다. 에이전트의 자율성과 인간 감독 사이의 균형을 유지한다.

### ④ Evaluator-Optimizer 패턴

하나의 LLM이 응답을 생성하고, 다른 LLM이 반복적으로 평가·피드백을 제공한다. 명확한 평가 기준과 반복 개선 가능성이 있을 때 효과적이다.

### ⑤ Entropy Management (가비지 컬렉션) 패턴

주기적 에이전트 실행으로 코드베이스 품질을 장기적으로 유지한다.

- 문서 불일치 감지·수정
- 아키텍처 제약 위반 식별
- 내부 코드 품질 모니터링

### ⑥ Context Window 관리 패턴

컨텍스트 윈도우가 가득 찰 때 자동으로 요약·압축(Context Compaction)하여 에이전트 연속성을 유지한다. Anthropic 멀티에이전트 연구 시스템은 **200,000 토큰 초과** 시 자동으로 메모리 시스템에 저장·복구한다.

## 7. 실제 구현 사례 — Stripe, OpenAI

### Stripe "Minions" 시스템

- **규모:** 주당 **1,300건 이상** PR 자동 머지, 인간이 작성한 코드 없음
- **구조:** 각 Minion은 격리된 컨테이너 + 코드베이스 신규 체크아웃으로 실행
- **안전 경계:**
  - 프로덕션 시스템 직접 접근 불가
  - main 브랜치 직접 푸시 불가
  - 정의된 범위 외 변경 불가
- **자기 수정 루프:** 코드 작성 → 테스트 → 오류 읽기 → 진단 → 수정 → 재테스트 (반복)
- **도구 관리:** MCP 서버 한 곳에 **400개 이상** 내부 도구 집중, 프로토콜 레벨 권한 강제

### OpenAI 내부 코드베이스 하네스 (100만+ 줄)

- 하네스 구축에 **5개월** 투자
- 세 가지 핵심 축:
  1. **컨텍스트 엔지니어링** — 리포지토리가 단일 진실의 원천(single source of truth)
  2. **아키텍처 제약** — Slack, Google Docs 정보는 컨텍스트에 포함시키지 않으면 에이전트에게 존재하지 않음
  3. **엔트로피 관리** — 주기적 에이전트 실행으로 코드 품질 유지

## 8. 2026년 하네스 엔지니어링 모범 사례

### 설계 원칙

```markdown
1. 단순하게 시작
   - 최소한의 제어 흐름
   - 복잡성은 성능 향상이 증명될 때만 추가

2. 모듈식 구축
   - 삭제·교체를 위해 설계
   - 모델 발전에 따라 하네스도 교체 가능하도록

3. 데이터 수집 우선
   - 궤적 로그(trajectory logs)가 훈련 반복의 경쟁 우위

4. 검증 강제화
   - "테스트를 삭제하거나 편집하는 것은 허용하지 않음"
   - 에이전트가 검증 단계를 우회할 수 없도록
```

### 보안 원칙

```markdown
1. 모든 것을 로깅
   - 코드 실행, 도구 호출, API 요청 전부

2. 이상 행동 탐지
   - 권한 거부, 정책 위반을 침해 지표로 추적

3. 최소 권한 & 단기 자격증명
   - 태스크별 제한된 범위의 임시 자격증명 발급

4. 네트워크 기본 차단
   - 이그레스 기본 차단, 필요 엔드포인트만 허용 목록
```

### 2026년 핵심 트렌드 요약

| 트렌드 | 내용 |
|------|------|
| **MCP 표준화** | 에이전트-도구 통합의 사실상 표준으로 자리잡음 (Anthropic, OpenAI, Google DeepMind 채택) |
| **하네스가 모델보다 중요** | 업계 초점이 모델 중심 → 인프라 중심으로 이동 |
| **Context Engineering** | 프롬프트 엔지니어링에서 컨텍스트 공학으로 패러다임 전환 |
| **Agentic Safety** | OWASP AI 에이전트 보안 Top 10 등 보안 표준 등장 |
| **Long-Running Agents** | 수 시간~수 일에 걸쳐 자율 실행되는 에이전트 상용화 |

하네스 엔지니어링은 AI 에이전트를 "실험실 밖"으로 꺼내는 기술이다. 모델 자체가 아무리 강력해도, 하네스 없이는 프로덕션에서 안전하게 동작하지 않는다. Anthropic, Stripe, OpenAI가 증명했듯 — **에이전트의 성패는 모델이 아니라 벽(하네스)이 결정한다.**
